## 第1章-1: OSIモデルとTCP/IPモデル

### 1.1 なぜOSIモデルを学ぶのか？

ネットワークのトラブルシューティング・サービス選択・セキュリティ設計のすべてにおいて、
「**どのレイヤーの問題か**」を特定する能力が必須です。
PCNEの試験では「このロードバランサーは何レイヤーで動作するか？」のような問いが直接出題されます。

### 1.2 OSI 7層モデルの完全解説

```text
OSI 参照モデル（Open Systems Interconnection）

┌─────────────────────────────────────────────────────────────────┐
│  Layer 7 │ アプリケーション層 │ HTTP, HTTPS, DNS, SMTP, FTP     │
│──────────┼───────────────────┼────────────────────────────────│
│  Layer 6 │ プレゼンテーション│ SSL/TLS暗号化, 文字コード変換    │
│──────────┼───────────────────┼────────────────────────────────│
│  Layer 5 │ セッション層      │ セッション確立・維持・終了        │
│──────────┼───────────────────┼────────────────────────────────│
│  Layer 4 │ トランスポート層  │ TCP（信頼性）, UDP（速度）        │
│──────────┼───────────────────┼────────────────────────────────│
│  Layer 3 │ ネットワーク層    │ IP, ICMP, ルーティング           │
│──────────┼───────────────────┼────────────────────────────────│
│  Layer 2 │ データリンク層    │ Ethernet, MAC, VLAN, ARP        │
│──────────┼───────────────────┼────────────────────────────────│
│  Layer 1 │ 物理層           │ ケーブル, 光ファイバー, 電波     │
└─────────────────────────────────────────────────────────────────┘
```

#### 各レイヤーの詳細解説

**Layer 1 物理層（Physical Layer）**

```text
役割: ビット（0と1）を物理的な信号に変換して伝送する

具体例:

  - 銅線ケーブル（CAT5e, CAT6）
  - 光ファイバーケーブル（シングルモード, マルチモード）
  - Wi-Fi の電波
  - Google Cloud の海底ケーブル（Curie, Dunant 等）

単位: ビット（bit）

トラブル例:

  - ケーブルの断線 → 接続自体が確立しない
  - コネクタの抜け → リンクダウン
  - 光ファイバーの曲がりすぎ → 信号減衰

```

**Layer 2 データリンク層（Data Link Layer）**

```text
役割: 同一ネットワーク（セグメント）内でのフレーム転送

重要な概念:
  MAC アドレス（Media Access Control Address）:

    - ネットワーク機器に固有の物理アドレス
    - 48ビット（6バイト）: 例 00:1A:2B:3C:4D:5E
    - 前半24bit: メーカー識別子（OUI）
    - 後半24bit: 機器固有番号

  ARP（Address Resolution Protocol）:

    - IPアドレス → MACアドレス の変換
    - 「このIPを持つ機器よ、MACを教えてくれ！」とブロードキャスト

    例:
    PC-A (10.0.0.1) が PC-B (10.0.0.2) に送信したい
    → ARP Request: "10.0.0.2 を持つ人は MAC を教えて"
    → ARP Reply: "私です。MAC: 00:1A:2B:3C:4D:5E"
    → PC-A が MACアドレスを使って直接フレームを送信

  VLAN（Virtual LAN）:

    - 物理的なスイッチを論理的に分割
    - 異なるVLAN間の通信にはルーターが必要

単位: フレーム（Frame）

GCPとの関連:

  - Cloud Interconnect の VLAN Attachment はオンプレミスネットワークと Google エッジ間のレイヤー 2 隣接性を提供し、物理/論理的な接続点として機能する
  - VPC 内の通信は「分散 L3 ルーティング（各ホストでの L3 転送）」が実態であり、仮想的な L2 スイッチングではない

```

**Layer 3 ネットワーク層（Network Layer）**

```text
役割: 異なるネットワーク間でのパケット転送（ルーティング）

重要な概念:
  IPアドレス（Internet Protocol Address）:

    - 論理的なアドレス（変更可能）
    - IPv4: 32ビット（例: 192.168.1.1）
    - IPv6: 128ビット（例: 2001:db8::1）

  ルーティング:

    - パケットの最適な経路を選択して転送
    - ルーティングテーブルに基づいて判断

  ICMP（Internet Control Message Protocol）:

    - ネットワークの診断・エラー報告
    - ping コマンドが使用する
    - traceroute がこれを利用

単位: パケット（Packet）

GCPとの関連:

  - Cloud Router はこのレイヤーで動作
  - VPC のルーティングテーブル
  - ファイアウォールルール（主にL3情報で判断）

```

**Layer 4 トランスポート層（Transport Layer）**

```text
役割: エンド・ツー・エンドの通信品質を管理

重要な概念:
  ポート番号（Port Number）:

    - アプリケーションを識別する番号（0〜65535）
    - Well-known ポート（0〜1023）: HTTP=80, HTTPS=443, SSH=22
    - 送信元ポート + 宛先ポートで通信を識別

  TCP（信頼性重視）:

    - 3ウェイハンドシェイクで接続確立
    - データの順序保証・再送制御
    - フロー制御・輻輳制御

  UDP（速度重視）:

    - 接続確立なし
    - 信頼性より速度を優先
    - DNS・動画ストリーミング・VoIP

単位: セグメント（TCP）/ データグラム（UDP）

GCPとの関連:

  - ロードバランサーの種類（L4 vs L7）の判断基準
  - ファイアウォールルールのポート指定
  - Cloud NAT のポート変換

```

**Layer 5-6-7 上位層**

```text
Layer 5 セッション層:

  - 通信セッションの管理
  - 接続・維持・終了・同期
  - TLS セッションの管理

Layer 6 プレゼンテーション層:

  - データの表現形式を変換
  - 暗号化・復号化（SSL/TLS）
  - 文字コード変換（ASCII・UTF-8等）
  - 圧縮・展開

Layer 7 アプリケーション層:

  - ユーザーが直接使うプロトコル
  - HTTP/HTTPS: Webブラウジング
  - DNS: 名前解決
  - SMTP: メール送信
  - FTP: ファイル転送
  - SSH: セキュアシェル

GCPとの関連:

  - L7 ロードバランサー（HTTP(S) LB）
  - Cloud DNS（L7 アプリケーション）
  - Cloud Armor（HTTPヘッダーの解析）
  - URL Map（L7のルーティング）

```

### 1.3 TCP/IP モデル（実用的な4層モデル）

```text
TCP/IP モデル（実際のインターネットで使用）

┌──────────────────────────────────────────────────────────┐
│  アプリケーション層  │ HTTP, HTTPS, DNS, SMTP, SSH      │
│                     │ = OSI Layer 5+6+7                │
├──────────────────────────────────────────────────────────┤
│  トランスポート層    │ TCP, UDP                         │
│                     │ = OSI Layer 4                    │
├──────────────────────────────────────────────────────────┤
│  インターネット層    │ IP, ICMP                         │
│                     │ = OSI Layer 3                    │
├──────────────────────────────────────────────────────────┤
│  ネットワーク接続層  │ Ethernet, Wi-Fi, ARP             │
│                     │ = OSI Layer 1+2                  │
└──────────────────────────────────────────────────────────┘
```

GCPサービスのOSIレイヤー対応表（試験頻出）:

## 第1章-2: IPアドレッシングとサブネッティング

### 2.1 IPv4 アドレスの構造

```text
IPv4 アドレス = 32ビットの数値

表現方法（ドット区切り10進数）:
  192.168.1.1
   ↑    ↑  ↑
  192  168  1  1  ← 各オクテット（8ビット）= 0〜255の値

2進数での表現:
  192.168.1.1
  = 11000000.10101000.00000001.00000001

IPアドレスの構成:
  ┌──────────────────────────────────────┐
  │  ネットワーク部  │  ホスト部         │
  │（どのネットワーク）│（そのネットワーク │
  │                  │  内の特定の機器）  │
  └──────────────────────────────────────┘

  例: 192.168.1.0/24
  → ネットワーク部: 192.168.1 (前24ビット)
  → ホスト部: 最後の8ビット（.0 〜 .255）
  → .0  = ネットワークアドレス（代表アドレス）
  → .255 = ブロードキャストアドレス（全機器への送信）
  → .1〜.254 = 使用可能なホストアドレス（254台）
```

### 2.2 CIDR（Classless Inter-Domain Routing）

```text
CIDR 表記法: IPアドレス/プレフィックス長

  /24 → サブネットマスク: 255.255.255.0 → ホスト数: 254
  /25 → サブネットマスク: 255.255.255.128 → ホスト数: 126
  /26 → サブネットマスク: 255.255.255.192 → ホスト数: 62
  /27 → サブネットマスク: 255.255.255.224 → ホスト数: 30
  /28 → サブネットマスク: 255.255.255.240 → ホスト数: 14
  /29 → サブネットマスク: 255.255.255.248 → ホスト数: 6
  /30 → サブネットマスク: 255.255.255.252 → ホスト数: 2

プレフィックス長の計算式:
  ホスト数 = 2^(32-プレフィックス長) - 2
  (/24 の場合: 2^(32-24) - 2 = 2^8 - 2 = 256 - 2 = 254)

CIDRで表現できる範囲:
  10.0.0.0/8  = 10.0.0.0 〜 10.255.255.255 (1677万台)
  10.0.0.0/16 = 10.0.0.0 〜 10.0.255.255  (65534台)
  10.0.0.0/24 = 10.0.0.0 〜 10.0.0.255   (254台)
```

### 2.3 プライベートIPアドレスとパブリックIPアドレス

```text
プライベートIPアドレス（RFC 1918）:
  インターネットには使用できない。LAN内専用のアドレス。

  10.0.0.0/8       = 10.0.0.0 〜 10.255.255.255
  172.16.0.0/12    = 172.16.0.0 〜 172.31.255.255
  192.168.0.0/16   = 192.168.0.0 〜 192.168.255.255

パブリックIPアドレス:
  上記以外のアドレス（インターネット上で一意）
  例: 8.8.8.8（Google Public DNS）
      34.100.x.x（Google Cloud のIPレンジ）

特殊なIPアドレス:
  127.0.0.0/8   = ループバック（自分自身）→ 127.0.0.1 (localhost)
  169.254.0.0/16 = リンクローカル（DHCP未取得時の自動設定）
  0.0.0.0/0      = デフォルトルート（すべての宛先）

GCPとの関連:
  VPCの内部IP: 必ずプライベートIPアドレス
  外部IP: Googleが管理するパブリックIPアドレス
  Private Google Access: プライベートIPのみで Google API にアクセス
```

### 2.4 サブネッティングの実践

```text
問題: 10.0.0.0/24 のネットワークを4つのサブネットに分割せよ

解答:
  /24 を4分割 → /26 (2ビット追加で4分割)

  サブネット1: 10.0.0.0/26   (10.0.0.0   〜 10.0.0.63)
  サブネット2: 10.0.0.64/26  (10.0.0.64  〜 10.0.0.127)
  サブネット3: 10.0.0.128/26 (10.0.0.128 〜 10.0.0.191)
  サブネット4: 10.0.0.192/26 (10.0.0.192 〜 10.0.0.255)

  各サブネットのホスト数: 2^(32-26) - 2 = 64 - 2 = 62台

PCNE 試験での応用:
  VPCサブネットを設計する際:

  - web用:    10.0.1.0/24  (254台まで対応)
  - app用:    10.0.2.0/24  (254台まで対応)
  - db用:     10.0.3.0/28  (14台まで対応・小規模DBクラスタ)
  - gke-pods: 10.100.0.0/16 (65534 Pod に対応)

  → 用途ごとに必要なIPアドレス数を計算してから設計する

GCPのGKEで特に重要:
  GKEクラスターはPodとServiceに大量のIPを必要とする
  /24 のサブネットでは Podが不足することが多い
  GKE用には/21〜/18 程度の余裕を持ったサブネットを準備する
```

### 2.5 IPv6 アドレスの基礎

```text
IPv6 = 128ビットのIPアドレス

表現方法（コロン区切り16進数）:
  2001:0db8:0000:0000:0000:0000:0000:0001
  = 2001:db8::1 （省略形）

なぜIPv6が必要か:
  IPv4: 約43億個 → 2011年に枯渇
  IPv6: 約340澗個（事実上無限）

IPv6の重要な特徴:

  - アドレス枯渇の解決
  - NAT不要（全デバイスがグローバルアドレスを持てる）
  - セキュリティ（IPsecを標準内包）
  - より効率的なルーティングヘッダー

GCPでのIPv6:

  - VPCサブネットでIPv6を有効化可能
  - グローバルLBはIPv6に対応（デュアルスタック）
  - IPv4/IPv6 デュアルスタックで両方サポート

```

### ✅ ベストプラクティス: IPアドレス設計

```yaml
PCNE 試験で問われる設計原則:

1. プライベートIPを使用する（RFC 1918）
   - VPC 内部通信は必ずプライベートIP
   - 外部IPは最小限に（セキュリティ上の原則）

2. オンプレミスとの重複を避ける
   - ハイブリッド接続を見越してCIDRを計画
   - 例: オンプレが 10.0.0.0/8 を使用 → VPC は 172.16.0.0/12 を使用

3. 拡張を見越した設計
   - 現在の台数 ✕ 4倍以上のアドレス空間を確保
   - GKEのPod/Service用に別のCIDRブロックを確保

4. サブネットの目的を明確に分ける
   - tier別分離（web/app/db）
   - 環境別分離（prod/stg/dev）
   - リージョン別に連番で管理（us: 10.1.x.x, eu: 10.2.x.x）

5. GCP特有の考慮事項
   - GCPはサブネットから最初の4IPと最後の1IPを予約する
   - 10.0.0.0/24 では実質250台のVMのみ作成可能

```

## 第1章-3: ルーティングの基礎

### 3.1 ルーティングとは何か？

```text
ルーティング = パケットを正しい宛先に届けるための「道案内」

郵便配達の比喩:
  ─────────────────────────────────────────
  郵便   = パケット（データ）
  郵便番号 = IPアドレス
  郵便局  = ルーター（Router）
  道順地図 = ルーティングテーブル
  ─────────────────────────────────────────

ルーターの動作:

  1. パケットを受信
  2. 宛先IPアドレスを確認
  3. ルーティングテーブルで一致するルートを探す
  4. 一致したルートの「次のホップ（Next Hop）」にパケットを転送
  5. 一致しない場合は「デフォルトルート（0.0.0.0/0）」を使用

```

### 3.2 ルーティングテーブルの読み方

```text
ルーティングテーブルの例:

宛先ネットワーク      サブネットマスク    ネクストホップ    インターフェース  メトリック
────────────────────────────────────────────────────────────────────────────────
192.168.1.0           /24               直結（Connected）  eth0           0
10.0.0.0              /16               192.168.1.1       eth0           10
172.16.0.0            /12               192.168.1.2       eth0           20
0.0.0.0               /0                192.168.1.254     eth0           100

読み方:
  → 宛先 192.168.1.5 → 直接接続のネットワーク（eth0で送信）
  → 宛先 10.0.5.100  → 192.168.1.1 に転送（10.0.0.0/16 に一致）
  → 宛先 8.8.8.8     → 192.168.1.254 に転送（デフォルトルート）

ロンゲストマッチルール（最も重要）:
  複数のルートが宛先IPに一致する場合、
  プレフィックス長が最も長い（より具体的な）ルートが優先される

  例: 宛先 10.0.1.5 の場合
    10.0.0.0/8  → 一致（広い範囲）
    10.0.0.0/16 → 一致（中程度）
    10.0.0.0/24 → 一致しない（.1.5 は範囲外）
  → 10.0.0.0/16 が選ばれる（最長マッチ）

GCPとの関連:
  VPC のルーティングテーブル = 同じ原則で動作
  Cloud Router でBGPを使って動的にルートを学習
```

### 3.3 静的ルーティングと動的ルーティング

```text
静的ルーティング（Static Routing）:

  - 管理者が手動でルーティングテーブルに登録
  - ルート変更時は手動更新が必要
  - シンプルだが管理コストが高い
  - 小規模・変化の少ないネットワーク向け

  GCPでの使用例:
  gcloud compute routes create my-static-route \
    --destination-range=192.168.0.0/24 \
    --network=my-vpc \
    --next-hop-vpn-tunnel=my-vpn-tunnel

動的ルーティング（Dynamic Routing）:

  - ルーティングプロトコルで自動的にルートを学習・更新
  - ネットワーク変更に自動対応
  - 大規模・変化の多いネットワーク向け
  - GCP では Cloud Router + BGP を使用

  主なルーティングプロトコル:
  IGP（内部）: OSPF, EIGRP, RIP（組織内部）
  EGP（外部）: BGP（インターネット・クラウドとの接続）

なぜGCPはBGPを使うのか:

  - スケーラビリティ（大規模ネットワーク対応）
  - 柔軟なルートポリシー制御
  - 標準プロトコル（ベンダー非依存）
  - HA VPN・Interconnect との完全な統合

```

### ✅ ベストプラクティス: ルーティング設計

```text

1. 動的ルーティング（BGP）を優先する

   → Cloud Router + HA VPN/Interconnect の組み合わせ
   → ルート変更の自動伝播でオペレーションコスト削減

2. デフォルトルートを意識した設計

   → 内部専用VPC ではデフォルトルートを削除
   → インターネットアクセスが不要なVMからの意図しない通信を防ぐ

3. ロンゲストマッチを活かした設計

   → 特定の宛先に向けた具体的なルートを追加することで
   → トラフィックを意図した経路に誘導できる

4. BGP MED（メトリック）でフェイルオーバーを制御

   → プライマリ経路: MED = 100
   → バックアップ経路: MED = 200
   → プライマリ障害時にバックアップに自動切り替わり
```

> 📎 参照: https://cloud.google.com/network-connectivity/docs/router/concepts/overview

## 第1章-4: TCP・UDP・ICMPプロトコル

### 4.1 TCP（Transmission Control Protocol）

```text
TCP の特徴: 信頼性のある接続型通信

3ウェイハンドシェイク（接続確立の儀式）:

  クライアント          サーバー
       │                  │
       │──── SYN ────────►│  ① 「接続したい」
       │◄─── SYN-ACK ─────│  ② 「わかった。私も接続準備OK」
       │──── ACK ────────►│  ③ 「わかった。接続確立」
       │                  │
       │◄──── データ交換 ──►│
       │                  │
  接続切断: 4ウェイハンドシェイク
       │──── FIN ────────►│
       │◄─── ACK ─────────│
       │◄─── FIN ─────────│
       │──── ACK ────────►│

TCPの重要な機能:
  ① シーケンス番号: データの順序を管理（順番通りに再構成）
  ② ACK（確認応答）: データ受信を送信元に通知
  ③ 再送制御: ACKが来ない場合は再送する
  ④ フロー制御: 受信側の処理能力に合わせて送信速度を調整
  ⑤ 輻輳制御: ネットワークの混雑を検知して送信量を調整

TCPを使うプロトコル（信頼性が必要なもの）:
  HTTP/HTTPS → Webブラウジング
  SSH        → セキュアシェル接続
  SMTP       → メール送信
  FTP        → ファイル転送

ファイアウォールとの関係:
  GCPファイアウォールはステートフル:
  → Ingressで tcp:80 を ALLOW すれば
  → その接続の戻り（Egress）は自動的に許可
  → つまり tcp:80 に明示的なEgressルール不要
```

### 4.2 UDP（User Datagram Protocol）

```text
UDP の特徴: 高速だが信頼性なし

クライアント          サーバー
     │                  │
     │──── データ ─────►│  ① そのまま送りっぱなし
     │──── データ ─────►│  ② ACKなし・順序保証なし
     │──── データ ─────►│  ③ 失われても再送しない
     │                  │

UDPを使うプロトコル（速度優先）:
  DNS      → 名前解決（遅延より速度が重要）
  DHCP     → IPアドレスの自動割り当て
  NTP      → 時刻同期
  VoIP     → 音声通話（少々の欠落より遅延が困る）
  動画配信  → リアルタイムストリーミング
  QUIC/HTTP3 → 次世代HTTP（UDPベースで高速化）
  DNS over HTTPS/TLS → 暗号化DNS

ファイアウォールの注意点:
  UDPはコネクションレス → セッション概念がない
  → GCPのステートフルFWは UDP でも
    最初のパケット方向を記憶して戻りを許可する
  → ただし UDP は TCP より制御が複雑
```

### 4.3 ICMP（Internet Control Message Protocol）

```text
ICMP の役割: ネットワークの診断・エラー通知

主なICMPメッセージ:
  Echo Request/Reply (Type 8/0):
    → ping コマンドで使用
    → 「応答してくれ」→「了解」

  Destination Unreachable (Type 3):
    → 宛先に届かない場合のエラー通知
    → 「その宛先には届けられない」

  Time Exceeded (Type 11):
    → TTL（生存時間）が0になった場合
    → traceroute コマンドで使用

TTL（Time To Live）:
  パケットが経由できるルーターの最大数
  ルーターを1つ通るたびに 1 減算
  0になるとルーターがパケットを破棄しICMPを返す

traceroute の仕組み:
  TTL=1 で送信 → 最初のルーターが破棄してICMP返信
  TTL=2 で送信 → 2番目のルーターが破棄してICMP返信
  ... → 宛先まで繰り返す → 経路のホップを可視化

GCPでの考慮:
  VPCファイアウォールでICMPを許可しないと ping が通らない
  デバッグ時は一時的にICMPを許可してから疎通確認
  本番環境では内部ネットワークのICMPのみ許可
```

### ✅ ベストプラクティス: プロトコル設計

```yaml
ファイアウォールルールの設計:
  HTTP/HTTPS(TCP): 外部→LB のみ開放
  SSH(TCP 22): IAPのIPレンジのみ (35.235.240.0/20)
  DNS(UDP/TCP 53): 内部ネットワークのみ
  ICMP: 内部ネットワークのみ（デバッグ用）
  すべての通信: 明示的に不要なものは DENY

注意点: UDP は接続確立がないため、より細かいフィルタリングが必要
  DoS攻撃でよく使われる → Cloud Armor でレート制限を設定
```

## 第1章-5: DNS（ドメイン名システム）

### 5.1 DNSとは何か？

```text
DNS = インターネットの「電話帳」

人間が覚えやすいドメイン名 ↔ 機械が理解できるIPアドレス を変換する

  example.com → 93.184.216.34

なぜDNSが必要か:
  IPアドレス（数字）は変更されることがある
  ドメイン名は変わらない
  → ドメイン名を変えずにIPを変更できる（サーバー移行に便利）
```

### 5.2 DNS の階層構造

```text
DNSの階層（ツリー構造）:

  . (ルート)
  ├── com （TLD: Top Level Domain）
  │   ├── example.com （権威NS: example.comを管理）
  │   │   ├── www.example.com  → 203.0.113.1
  │   │   ├── api.example.com  → 203.0.113.2
  │   │   └── mail.example.com → 203.0.113.3
  │   └── google.com
  ├── jp
  │   └── co.jp
  └── internal （プライベートDNS）

DNS 解決の流れ:

  [ユーザーのブラウザ]
       ↓ www.example.com を知りたい
  [リゾルバー（ISP or 8.8.8.8）]
       ↓ キャッシュになければ問い合わせ
  [ルートDNSサーバー]
       ↓ "com の NS は .com サーバーです"
  [.com の DNS サーバー]
       ↓ "example.com の NS は ns1.example.com です"
  [example.com の権威DNSサーバー]
       ↓ "www.example.com = 203.0.113.1 です"
  [リゾルバーがキャッシュして回答]
       ↓
  [ブラウザが 203.0.113.1 に接続]
```

### 5.3 DNS レコードタイプ

```text
主要な DNS レコードタイプ（試験頻出）:

A レコード:
  ドメイン名 → IPv4 アドレスへの変換
  example.com  IN A  203.0.113.1

AAAA レコード:
  ドメイン名 → IPv6 アドレスへの変換
  example.com  IN AAAA  2001:db8::1

CNAME レコード:
  別名（エイリアス）を設定
  www.example.com  IN CNAME  example.com
  → www.example.com を解決すると example.com のIPを返す
  ※ Zone Apex（example.com 自体）にはCNAMEを設定できない

MX レコード:
  メールサーバーを指定
  example.com  IN MX 10 mail.example.com

NS レコード:
  権威DNSサーバーを指定
  example.com  IN NS ns1.example.com

PTR レコード:
  IPアドレス → ドメイン名への逆引き変換
  1.113.0.203.in-addr.arpa  IN PTR example.com

TXT レコード:
  テキスト情報（SPF・DKIM・ドメイン認証）
  example.com  IN TXT "v=spf1 include:google.com ~all"

SOA レコード:
  ゾーンの管理情報（権威サーバー・TTL等）

GCPとの関連:
  Cloud DNS で上記全レコードを管理
  LBのIPをAレコードで登録
  ドメイン認証はTXTレコードで実施
```

### 5.4 TTL（Time To Live）とDNSキャッシュ

```text
TTL = DNSレコードのキャッシュ有効期間（秒単位）

TTL 3600  → 1時間キャッシュ
TTL 300   → 5分キャッシュ
TTL 86400 → 24時間キャッシュ

TTL の設計指針:
  変更予定なしの安定したレコード:
    → TTL: 86400（24時間）→ DNSサーバーへの問い合わせ削減

  変更予定があるレコード（移行前）:
    → TTL: 300（5分）→ 変更が素早く伝播する

  移行手順:
    ① 現在 TTL: 86400 を 300 に変更
    ② 最低 86400秒（24時間）待つ（古いキャッシュが消えるまで）
    ③ IPアドレスを新しいサーバーのIPに変更
    ④ 移行完了後 TTL を 86400 に戻す

Cloud DNS での考慮:
  Google Cloud LB のIPは変わらない（静的予約IP使用時）
  → 安定したTTLを設定可能
  Cloud Run/Functions のURLは変わらない
  → CNAMEで管理してTTLの影響を最小化
```

### ✅ ベストプラクティス: DNS 設計

```yaml
Cloud DNS のベストプラクティス:

1. プライベートゾーンで内部サービスを管理

   内部サービスにはプライベートDNSを使用
   → internal.example.com → 10.0.0.5（内部IPに解決）

2. オンプレとのDNS統合

   転送ゾーン: GCP → オンプレのDNSに問い合わせ転送
   インバウンド: オンプレ → Cloud DNS に問い合わせ

3. DNSSECで公開ゾーンを保護

   DNSのキャッシュポイズニング攻撃を防ぐ

4. TTL 設計

   本番環境の安定したレコード: TTL 300〜3600
   移行直前: TTL 60〜300 に短縮してから変更

5. Cloud DNS のログ記録を有効化

   名前解決のログで不審な通信を検知
```

> 📎 参照: https://cloud.google.com/dns/docs/overview
> 📎 参照: https://cloud.google.com/dns/docs/best-practices

## 第1章-6: NATとPATの仕組み

### 6.1 NAT（Network Address Translation）とは？

```text
NAT = プライベートIPとパブリックIPを変換する仕組み

なぜNATが必要か:
  IPv4アドレスは約43億個しかない
  全世界の機器に割り当てるには足りない
  → 組織内部はプライベートIPを使用
  → 外部との通信時のみNATでパブリックIPに変換

NATの動作:

  ┌── 内部ネットワーク ────────────────────────────────────────┐
  │  PC-A  10.0.0.1:52000                                    │
  │  PC-B  10.0.0.2:54000 ─→ NAT機器 ─→ 外部IP: 203.0.113.1│
  │  PC-C  10.0.0.3:56000                                    │
  └─────────────────────────────────────────────────────────┘

NATテーブル（変換テーブル）:
  内部IP:ポート          外部IP:ポート          宛先
  10.0.0.1:52000   →   203.0.113.1:10001  →  8.8.8.8:80
  10.0.0.2:54000   →   203.0.113.1:10002  →  8.8.8.8:80
  10.0.0.3:56000   →   203.0.113.1:10003  →  8.8.8.8:80

戻りのパケット:
  8.8.8.8:80 → 203.0.113.1:10001
  NATテーブルで変換: → 10.0.0.1:52000 に転送
```

### 6.2 NATの種類

```text
NATの種類:

  1. Static NAT（1対1変換）:

     プライベートIP 1つ ↔ パブリックIP 1つ を固定で変換
     用途: 外部に公開するサーバー（Webサーバー等）
     GCPでの対応: 外部IPアドレスを直接VMに割り当て

  2. Dynamic NAT:

     複数のプライベートIP → IPプールから動的に変換
     外部からの接続は不可（インバウンドはNATテーブルに記録なし）

  3. PAT / NAPT（Port Address Translation）:

     複数のプライベートIP → 1つのパブリックIP（ポートで識別）
     最もよく使われる方式
     GCPでの対応: Cloud NAT → まさにこの仕組み

Cloud NAT の動作:
  VM (10.0.0.5) がインターネットにアクセスしたい
       ↓
  Cloud NAT Gateway が受信
       ↓ 外部IP + ポートに変換
  203.0.113.1:10001 → インターネット

  重要: インバウンドは不可（外からVMへの接続はできない）
       これがCloud NATのセキュリティの鍵
```

### 6.3 ポート番号とNATの関係

```text
ポートの分類:

  Well-known ポート (0〜1023):
    サービスが使う固定ポート
    80  = HTTP
    443 = HTTPS
    22  = SSH
    53  = DNS
    25  = SMTP

  エフェメラルポート (1024〜65535):
    クライアントが通信時に一時的に使うポート
    OS が自動割り当て
    Linuxのデフォルト: 32768〜60999

Cloud NAT のポート消費:
  VM 1台がインターネットに接続するたびに
  NATポートを1つ消費する

  例: VM 10台が各100本のHTTPS接続を持つ場合
  → 10 × 100 = 1000 ポートを消費

  ポート枯渇の対策:
  → 複数の外部IPを追加（各IPで65535ポート使用可能）
  → Dynamic Port Allocation を有効化
  → Cloud NAT のログでエラー検知
```

### ✅ ベストプラクティス: NAT 設計

```text

1. Cloud NAT はアウトバウンドのみ（セキュリティ設計に活用）

   → インバウンドは LB・IAP・VPN 経由が原則

2. 外部IPを持つVMを最小限にする

   → 全VMに外部IP = セキュリティリスク増大
   → 外部IPなし + Cloud NAT がベスト

3. 静的外部IPを予約する（本番環境）

   → LBやNATに使うIPは静的予約してIPが変わらないように

4. NATログを有効化してポート枯渇を監視

   gcloud compute routers nats update my-nat \
     --enable-logging \
     --log-filter=ERRORS_ONLY

5. DynamicPortAllocationで効率化

   → 負荷に応じて自動的にポート数を調整
```

> 📎 参照: https://cloud.google.com/nat/docs/overview
> 📎 参照: https://cloud.google.com/nat/docs/ports-and-addresses

## 第1章-7: VLANとネットワーク分割

### 7.1 VLAN（Virtual Local Area Network）とは？

```text
VLAN = 1つの物理スイッチを複数の論理ネットワークに分割する技術

VLANなし（フラットネットワーク）:
  ┌── スイッチ ────────────────────────────────────────┐
  │  [PC-A] [PC-B] [PC-C] [サーバー] [プリンター]      │
  │  全てが同じネットワーク → 全員が全員と通信できる     │
  └─────────────────────────────────────────────────────┘
  問題: セキュリティ境界がない、ブロードキャスト増大

VLANあり（ネットワーク分割）:
  ┌── スイッチ ───────────────────────────────────────┐
  │  VLAN 10（業務用）:  [PC-A] [PC-B] [サーバー]     │
  │  VLAN 20（ゲスト）:  [ゲストPC-1] [ゲストPC-2]    │
  │  VLAN 30（管理）:    [管理PC] [ルーター]           │
  │                                                    │
  │  → VLAN 10と20は直接通信できない                   │
  │  → ルーター（L3機器）を経由しないと通信不可         │
  └─────────────────────────────────────────────────────┘
```

### 7.2 VLAN の動作原理

```text
VLANタグ（IEEE 802.1Q）:
  Ethernetフレームにタグを付けてVLANを識別

  通常のEthernetフレーム:
  [宛先MAC][送信元MAC][タイプ][データ][FCS]

  802.1Q タグ付きフレーム:
  [宛先MAC][送信元MAC][0x8100][VLAN ID(12bit)][タイプ][データ][FCS]
                          ↑                      ↑
                    VLANタグの識別子          VLAN番号(1〜4094)

スイッチポートの種類:
  Access ポート:

    - 1つのVLANのみ
    - PCやサーバーを接続
    - タグなしフレームで通信

  Trunk ポート:

    - 複数のVLANを1本のケーブルで転送
    - スイッチ間・スイッチ-ルーター間に使用
    - タグ付きフレームで通信

GCPとの関連:
  Cloud Interconnect の VLAN Attachment:
    物理回線を論理的に分割してVPCに接続
    Dedicated Interconnect → 最大10Gbpsの物理回線
    VLAN Attachment 1 → VPC-A に接続
    VLAN Attachment 2 → VPC-B に接続
    → 1本の回線で複数VPCに分割接続できる！
```

### ✅ ベストプラクティス: ネットワーク分割

```text
セキュリティとパフォーマンスのための分割設計:

1. 用途別に分離する

   本番: VLAN 100 / 開発: VLAN 200 / 管理: VLAN 300

2. インターVLAN通信はルーターで制御

   異なるVLAN間の通信 = 明示的にファイアウォールで許可

3. GCPでの実装:

   VLAN分離 → VPCサブネット分離（機能的に同等）
   インターVLAN通信制御 → VPCファイアウォールルール
   Trunk port → Cloud Interconnect VLAN Attachment
```

> 📎 参照: https://cloud.google.com/network-connectivity/docs/interconnect/concepts/vlan-attachments

## 第2章-8: ファイアウォールの基礎

### 8.1 ファイアウォールの種類

```text
ファイアウォールの進化:

世代1: パケットフィルタリング（ステートレス）

  - IP・ポートのみで判断
  - 各パケットを独立して評価
  - 戻りのパケットも明示的に許可が必要
  - 高速だが制御が粗い

世代2: ステートフルファイアウォール

  - セッション（接続状態）を追跡
  - 許可した接続の戻りは自動許可
  - GCPのVPCファイアウォールはこれ
  - 一般的なエンタープライズ用途に最適

世代3: 次世代ファイアウォール（NGFW）

  - L7（アプリケーション層）での判断
  - URLフィルタリング
  - アプリケーション識別
  - IPS/IDSとの統合
  - GCPでは Cloud Armor がL7機能を提供

世代4: クラウドネイティブFW

  - API ベースで集中管理
  - スケールアウトに対応
  - GCPの階層型ファイアウォールポリシー

```

### 8.2 ステートフルとステートレスの違い

```text
ステートレスFW（パケットフィルタ）:

  状況: クライアントが Webサーバーの 80番ポートにアクセス

  必要なルール（双方向に設定が必要）:
  ① ALLOW: src=クライアントIP, dst=サーバーIP, dport=80
  ② ALLOW: src=サーバーIP, dst=クライアントIP, sport=80

  → 双方向に明示的にルールが必要（管理が複雑）

ステートフルFW（GCPのVPCファイアウォール）:

  必要なルール（一方向だけでOK）:
  ① ALLOW: ingress, dst=サーバーIP, dport=80

  → ①のルールがあれば、その接続の戻りは自動で許可される
  → 管理がシンプル

セッションテーブルの仕組み:
  クライアントが 10.0.0.1:54321 → 10.0.0.2:80 で接続

  セッションテーブルに記録:
  [10.0.0.1:54321 ↔ 10.0.0.2:80] ESTABLISHED

  10.0.0.2:80 → 10.0.0.1:54321 のパケットが来たら:
  → テーブルで確認 → 既存セッションの戻り → 自動許可
```

### 8.3 ファイアウォールルールの設計

```text
デフォルトポリシーの考え方:

  デフォルト許可（Allow by Default）:
    → 明示的に拒否したもの以外は全て許可
    → 設定は簡単だが危険（想定外の通信が通ってしまう）

  デフォルト拒否（Deny by Default）← 推奨:
    → 明示的に許可したもの以外は全て拒否
    → GCP の VPC ファイアウォールのデフォルト
    → ホワイトリスト方式で安全

GCPのデフォルトルール:
  ① 全 Egress（外向き）: ALLOW  優先度 65535（最低優先度）
  ② 全 Ingress（内向き）: DENY   優先度 65535（最低優先度）

  → デフォルトではIngressは全て拒否
  → 必要なIngress通信を明示的に許可ルールで追加

DMZ（非武装地帯）設計:
  ┌── インターネット ──────────────────────────────────┐
  │           ↓ HTTP/HTTPS のみ許可                    │
  │  ┌── DMZ（Web層） ───────────────────────────┐   │
  │  │  [Webサーバー]                             │   │
  │  │           ↓ アプリポートのみ許可             │   │
  │  │  ┌── 内部ネットワーク（App層）──────────┐   │   │
  │  │  │  [APPサーバー]                       │   │   │
  │  │  │           ↓ DBポートのみ許可          │   │   │
  │  │  │  ┌── 最内部（DB層）───────────┐     │   │   │
  │  │  │  │  [DBサーバー]              │     │   │   │
  │  │  │  └──────────────────────────┘     │   │   │
  │  │  └──────────────────────────────────┘   │   │
  │  └──────────────────────────────────────────┘   │
  └──────────────────────────────────────────────────┘

GCPでの実装:
  web-tier サブネット → LBのIPからHTTP/HTTPS許可
  app-tier サブネット → web-tierのサービスアカウントからAppPort許可
  db-tier サブネット  → app-tierのサービスアカウントからDB Port許可
```

### ✅ ベストプラクティス: ファイアウォール

```text

1. デフォルト拒否ポリシー（最重要）
2. 最小権限の原則: 必要なポートのみ開放
3. SSH/RDPは直接開放しない → IAP を使用
4. 管理用ポートは特定IPからのみ許可
5. ファイアウォールログを必ず有効化
6. 定期的にルールを棚卸しして不要ルールを削除
7. 組織レベルの階層型ポリシーで基本ルールを統一

```

## 第2章-9: VPNと暗号化

### 9.1 VPN（Virtual Private Network）の基礎

```text
VPN = インターネット上にプライベートな「トンネル」を作る技術

なぜVPNが必要か:
  インターネット上の通信は盗聴・改ざんのリスクがある
  → 暗号化したトンネルで安全に通信

  拠点A ──── 暗号化トンネル（インターネット経由）──── 拠点B
  10.0.0.0/24                                         10.1.0.0/24

VPNの種類:
  ① Site-to-Site VPN（拠点間VPN）:
     オフィスやデータセンター同士を常時接続
     → Cloud VPN がこれ

  ② Remote Access VPN（リモートアクセスVPN）:
     個人のPCからVPNサーバーに接続
     → テレワークで使う一般的なVPN

  ③ SSL-VPN:
     ブラウザベースのVPN
     → Google の IAP も類似の概念
```

### 9.2 IPsec（IP Security）の仕組み

```text
IPsec = VPNで最も広く使われる暗号化プロトコル

IPsecの2つのモード:
  トランスポートモード:
    元のIPヘッダーを保持してデータ部のみを暗号化
    エンドツーエンドの暗号化に使用

  トンネルモード:
    元のパケット全体を新しいIPヘッダーで包む（カプセル化）
    Site-to-Site VPN に使用（Cloud VPNも使用）

    ┌── 元のパケット ────────────────────────────┐
    │  [IPヘッダー: src=10.0.0.1][データ]         │
    └─────────────────────────────────────────────┘
    ↓ IPsecトンネルモードでカプセル化
    ┌── 暗号化されたパケット ────────────────────────┐
    │  [新IPヘッダー][ESP][暗号化{IPヘッダー+データ}] │
    └───────────────────────────────────────────────┘

IKE（Internet Key Exchange）:
  VPN接続時の鍵交換プロトコル

  IKEv1（旧世代）: 複雑・脆弱性あり → 非推奨
  IKEv2（現代標準）: シンプル・安全・高速 → 推奨

  Cloud VPN で IKEv2 を推奨する理由:

  - より安全な暗号化アルゴリズム
  - 接続の再確立が高速
  - モバイルクライアントにも対応

IKEの接続確立フェーズ:
  Phase 1: VPNゲートウェイ同士の認証と安全なチャネル確立
    → Pre-Shared Key（事前共有鍵）で認証
    → または証明書（PKI）で認証

  Phase 2: 実際のデータ転送用のセキュリティアソシエーション確立
    → 使用する暗号化アルゴリズムをネゴシエーション
    → 対称鍵を交換
```

### 9.3 暗号化アルゴリズムの基礎

```text
対称鍵暗号化（共通鍵暗号化）:
  同じ鍵で暗号化・復号化
  高速 → 大量データの暗号化に使用

  AES（Advanced Encryption Standard）:
    現代の標準的な対称鍵暗号
    AES-128: 鍵長128bit → 標準的
    AES-256: 鍵長256bit → より高セキュリティ

  課題: 鍵の配送問題（どうやって安全に鍵を共有するか）

非対称鍵暗号化（公開鍵暗号化）:
  公開鍵で暗号化 → 秘密鍵で復号化
  鍵の配送問題を解決
  低速 → 鍵交換に使用

  RSA: 広く使われる公開鍵暗号
  楕円曲線暗号（ECDSA）: 同じ強度をより短い鍵で実現

VPNでの組み合わせ:

  1. 非対称鍵で「対称鍵」を安全に交換（IKEフェーズ1）
  2. 交換した対称鍵でデータを暗号化（IKEフェーズ2）

  → 高セキュリティ + 高速処理を両立

Cloud VPN の推奨暗号スイート:
  IKE: AES-256-GCM with SHA-256 and ECDHE
  ESP: AES-256-GCM
```

### ✅ ベストプラクティス: VPN設計

```text

1. Cloud VPN は HA VPN を使用（99.99% SLA）

   → 2本のトンネルで冗長化
   → Classic VPN は新規構築で使用しない

2. IKEv2 を使用（IKEv1 は使わない）

3. 強力な事前共有鍵（PSK）を使用

   → 自動生成の長いランダム文字列（32文字以上）
   → Secret Manager で安全に保管

4. BGPで動的ルーティング

   → 静的ルートは管理コストが高い

5. 帯域要件がある場合は Interconnect を検討

   VPN: 最大3Gbps/トンネル
   Interconnect: 10G〜100G
```

> 📎 参照: https://cloud.google.com/network-connectivity/docs/vpn/concepts/overview
> 📎 参照: https://cloud.google.com/network-connectivity/docs/vpn/concepts/choosing-networks-routing

## 第2章-10: TLS/SSL と証明書

### 10.1 TLS/SSL の役割

```text
TLS（Transport Layer Security）:
  以前はSSL（Secure Sockets Layer）と呼ばれていた
  現在は TLS 1.2 / TLS 1.3 が主流

TLS が提供する3つの保証:

  1. 機密性（暗号化）: 通信内容を第三者が盗聴できない
  2. 完全性（改ざん検知）: 通信途中でデータが改ざんされていない
  3. 認証（なりすまし防止）: 接続先が本物であることを確認

HTTPSは HTTP + TLS:
  HTTP:  平文通信（パスワードが盗聴される危険）
  HTTPS: TLSで暗号化された通信（安全）

TLSハンドシェイクの流れ:
  ① Client Hello: クライアントが対応TLSバージョン・暗号スイートを通知
  ② Server Hello: サーバーが使用する暗号スイートを選択
  ③ Certificate:  サーバーが証明書を送信
  ④ Key Exchange: 安全な鍵交換（DHE/ECDHE）
  ⑤ Finished:     ハンドシェイク完了・暗号化通信開始
```

### 10.2 デジタル証明書の仕組み

```text
デジタル証明書 = 「このサーバーは本物です」という証明書

証明書に含まれる情報:

  - ドメイン名（Subject）: example.com
  - 公開鍵: RSA 2048bit など
  - 有効期限: 2025/01/01 〜 2026/01/01
  - 発行者（CA）: DigiCert / Let's Encrypt 等
  - CAのデジタル署名: 改ざん検知

認証局（CA: Certificate Authority）:
  証明書を発行・検証する信頼された機関

証明書チェーン（信頼の連鎖）:
  ┌── ルートCA（信頼のアンカー）────────────────┐
  │  DigiCert Root CA                           │
  └──────┬──────────────────────────────────────┘
         │ 署名
  ┌──────┴──────────────────────────────────────┐
  │  中間CA                                      │
  │  DigiCert SHA2 Secure Server CA             │
  └──────┬──────────────────────────────────────┘
         │ 署名
  ┌──────┴──────────────────────────────────────┐
  │  サーバー証明書                               │
  │  example.com                                 │
  └─────────────────────────────────────────────┘

ブラウザの動作:

  1. サーバー証明書を受信
  2. 証明書の署名を確認（CAの公開鍵で）
  3. CAの証明書をさらに確認
  4. ルートCAまで辿って信頼を確認
  5. ドメイン名が一致するか確認

GCPとの関連:
  Google マネージド証明書: Googleが自動的に取得・更新
  → Let's Encrypt を使って無料で証明書を発行
  → 90日ごとに自動更新
  → LBに適用するだけ（管理不要）
```

### 10.3 TLS バージョンとセキュリティ

```text
TLSバージョンの変遷（試験に関連）:

  SSL 3.0: 廃止済み（POODLE攻撃で脆弱性判明）
  TLS 1.0: 廃止推奨（BEAST攻撃）
  TLS 1.1: 廃止推奨
  TLS 1.2: 現在の標準（推奨）
  TLS 1.3: 最新・最高速・最高セキュリティ（推奨）

Cloud LBのSSLポリシー:
  Compatible: TLS 1.0以上（最も広い互換性、安全性低い）
  Modern:     TLS 1.2以上（推奨・一般的な選択）
  Restricted: TLS 1.2のみ（PCI DSS等のコンプライアンス向け）
  Custom:     手動で暗号スイートを指定

推奨設定:
  一般的なWebサービス → Modern (TLS 1.2以上)
  金融・医療等コンプライアンス → Restricted (TLS 1.2のみ)
  最新ブラウザのみ対象 → TLS 1.3 をCustomで指定

暗号スイートの例（TLS 1.2）:
  TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
  ↑ECDHE: 鍵交換（完全前方秘匿性）
       ↑RSA: 認証
              ↑AES-256-GCM: 対称鍵暗号化
                            ↑SHA-384: メッセージ認証
```

### ✅ ベストプラクティス: TLS/証明書

```text

1. TLS 1.2 以上を必須化（TLS 1.0/1.1 は無効化）
2. Google マネージド証明書を使用（自動更新・コスト削減）
3. HTTP→HTTPSリダイレクトを設定（常時HTTPS化）
4. HSTSヘッダーを設定（HTTPSを強制するブラウザ機能）
5. 証明書の有効期限を監視（30日前にアラート）
6. HPKP（証明書ピンニング）は慎重に使用（誤設定リスクあり）
7. ワイルドカード証明書（*.example.com）は利用範囲を限定

```

## 第2章-11: ゼロトラストセキュリティ

### 11.1 従来のセキュリティモデルの問題点

```text
従来のモデル：境界型セキュリティ（Castle and Moat）

  インターネット (信頼しない)
  ──────────────────────────
  [ファイアウォール（城壁）]
  ──────────────────────────
  社内ネットワーク (すべて信頼)
  [PC-A] [PC-B] [サーバー] [DB]

問題点:
  ① VPN で社内に入ってしまえば全てにアクセスできてしまう
  ② マルウェアが1台に侵入すると横移動（ラテラルムーブメント）が容易
  ③ リモートワーク・クラウド化で「境界」が曖昧になった
  ④ 内部の悪意あるユーザーを防げない

2020年代の現実:

  - 社員はカフェ・自宅・出張先から接続する
  - データはクラウドに存在する
  - デバイスは会社支給と個人端末が混在する

  → 「社内 = 安全」という前提が崩壊している
```

### 11.2 ゼロトラストの原則

```text
ゼロトラスト = 「決して信頼せず、常に検証せよ」
               "Never Trust, Always Verify"

ゼロトラストの6原則:

  1. すべてのアクセスをデフォルト拒否
  2. ユーザー・デバイス・アクセス先を毎回認証・認可
  3. アクセスは最小権限に限定
  4. ネットワーク位置に関わらず同じポリシーを適用
  5. すべてのアクセスをログ・監視
  6. マイクロセグメンテーション（細かい境界設定）

認証・認可の判断要素:
  ユーザーアイデンティティ: Googleアカウント・MFA
  デバイス状態: 最新OS・セキュリティパッチ適用済みか
  アクセス場所: IP・地理情報
  アクセス対象: 何のリソースか（機密度）
  時間: 業務時間内か

Google の BeyondCorp:
  Google が社内で実装したゼロトラストモデル
  → Cloud の IAP（Identity-Aware Proxy）として公開
  → VPNなしで安全なアクセスを実現
```

### 11.3 GCPでのゼロトラスト実装

```text
GCP のゼロトラストコンポーネント:

  ① IAP（Identity-Aware Proxy）:
     Webアプリ・VM へのアクセスをGoogle IDで認証
     デバイス状態・場所などを条件にアクセス制御

  ② Access Context Manager:
     アクセスポリシーを定義（IPレンジ・デバイス状態等）
     アクセスレベルをリソースに適用

  ③ VPC Service Controls:
     GCPのAPIサービスへのアクセスを境界で制限
     データ漏洩防止（DLP）

  ④ Chronicle (SIEM):
     全アクセスのログ収集・分析
     異常なアクセスパターンを検知

  ⑤ BeyondCorp Enterprise:
     上記全てを統合したゼロトラストソリューション

実装のステップ:
  Step 1: 資産の可視化（どこに何があるか把握）
  Step 2: リスク評価（機密データの特定）
  Step 3: マイクロセグメンテーション（VPC・サブネット分割）
  Step 4: アイデンティティ管理の強化（MFA必須化）
  Step 5: 継続的な監視・改善
```

### ✅ ベストプラクティス: ゼロトラスト

```text

1. IAPでSSH/RDP・内部Webアプリを保護（VPN廃止を目指す）
2. すべての通信を暗号化（mTLS for サービス間通信）
3. 最小権限の原則を徹底（IAMロール・VPCファイアウォール）
4. 全アクセスログを収集してSIEMで分析
5. デバイス管理（Endpoint Verification）を導入
6. 定期的なアクセス権限の棚卸し（不要な権限を削除）

```

> 📎 参照: https://cloud.google.com/beyondcorp-enterprise/docs/overview

## 第3章-12: ロードバランシングの基礎

### 12.1 ロードバランサーが必要な理由

```text
問題: 1台のサーバーでは限界がある

  1台のサーバーで対応可能な同時接続: 数百〜数千
  Googleの検索: 毎秒数十億のリクエスト
  人気のECサイトのセール: 通常の100倍のトラフィック

  解決策: 複数サーバーにトラフィックを分散する

  [ユーザー A]  ─→
  [ユーザー B]  ─→  [ロードバランサー]  ─→  [サーバー 1]
  [ユーザー C]  ─→                      ─→  [サーバー 2]
  [ユーザー D]  ─→                      ─→  [サーバー 3]

ロードバランサーが提供する機能:
  ① トラフィック分散: 複数サーバーに均等に振り分け
  ② 健全性監視: 障害サーバーを自動で除外
  ③ スケーラビリティ: サーバーを増減して容量を調整
  ④ 高可用性: サーバー障害時も通信を継続
  ⑤ SSL終端: 証明書管理を一元化
  ⑥ セッション維持: 同じユーザーを同じサーバーに向ける
```

### 12.2 負荷分散アルゴリズム

```text
主要な負荷分散アルゴリズム:

① ラウンドロビン（Round Robin）:
  順番に各サーバーにリクエストを割り当て
  最もシンプル・均等分散
  問題: サーバーの能力差を考慮しない

  リクエスト1 → サーバーA
  リクエスト2 → サーバーB
  リクエスト3 → サーバーC
  リクエスト4 → サーバーA  ← 繰り返し

② 加重ラウンドロビン（Weighted Round Robin）:
  サーバーの能力に応じて重みを設定
  サーバーA（性能高）: weight=3
  サーバーB（性能普通）: weight=1
  → A:A:A:B:A:A:A:B... の順で振り分け

③ 最少接続数（Least Connections）:
  現在の接続数が最も少ないサーバーに割り当て
  処理時間の長いリクエストが偏らない
  GCPのロードバランサーが使用する方式の一つ

④ IPハッシュ（IP Hash）:
  クライアントのIPをハッシュして同じサーバーに固定
  セッション維持が必要な場合に使用
  問題: サーバー追加/削除時に大量の再ハッシュが発生

⑤ ランダム（Random）:
  ランダムにサーバーを選択
  シンプルで大規模時に効果的

GCPのロードバランサーで使用:
  セッションアフィニティ（スティッキーセッション）:
  → IPアドレス・Cookieで同じバックエンドに固定可能
  → ステートフルアプリケーションに使用
```

### 12.3 L4 vs L7 ロードバランサー

```text
L4 ロードバランサー（トランスポート層）:

  判断基準: IPアドレス + ポート番号

  特徴:
  ✓ 高速（コンテンツを解析しない）
  ✓ あらゆるTCP/UDPトラフィックに対応
  ✓ クライアントIPをバックエンドに保持（パススルー）
  ✗ URLパスでのルーティング不可
  ✗ HTTP ヘッダーの解析不可

  使用例:

  - ゲームサーバー（TCP/UDP）
  - カスタムプロトコル
  - クライアントIPが必要なアプリ

L7 ロードバランサー（アプリケーション層）:

  判断基準: URL・HTTPヘッダー・Cookieなど

  特徴:
  ✓ URLパスで異なるサービスに振り分け可能
  ✓ HTTPヘッダーの書き換え・追加
  ✓ SSL/TLS の終端（証明書をLBで管理）
  ✓ Webアプリに最適
  ✗ HTTPSの中身を見るためSSL終端が必要
  ✗ L4より若干遅い（コンテンツ解析のオーバーヘッド）

  使用例:

  - Webアプリ（最も一般的）
  - マイクロサービス間のルーティング
  - A/Bテスト（特定URLを別バックエンドに）

具体例（URL マップ）:
  example.com/api/*  → APIサーバー群（8コア, 8台）
  example.com/images/* → Cloud Storage（CDN配信）
  example.com/*       → Webサーバー群（4コア, 4台）
```

### ✅ ベストプラクティス: ロードバランシング

```text

1. Webアプリは原則 L7（HTTP(S)）ロードバランサーを使用
2. グローバル配信が必要 → Global External HTTP(S) LB
3. Cloud Armor を組み合わせてセキュリティを強化
4. Cloud CDN を有効化してキャッシュでパフォーマンス向上
5. ヘルスチェックは適切に設定（Unhealthy Thresholdを低く）
6. バックエンドは複数ゾーンに分散（可用性向上）
7. セッションアフィニティはステートフルアプリのみに使用
8. Managed SSL証明書で証明書管理を自動化

```

## 第3章-13: BGPとルーティングプロトコル

### 13.1 BGP（Border Gateway Protocol）の基礎

```text
BGPとは = インターネットの「郵便局ネットワーク」

  インターネット = 無数の自律システム（AS）の集合
  AS（Autonomous System）= 独自のルーティングポリシーを持つネットワーク
  AS番号（ASN）= 各ASに割り当てられた固有番号

  Google: AS15169
  Amazon: AS16509
  NTT:    AS2914

BGPの役割:
  AS間でルーティング情報（どのIPレンジがどのASにあるか）を交換する

iBGP vs eBGP:
  iBGP（Internal BGP）: 同じAS内のルーター間
  eBGP（External BGP）: 異なるAS間

  Cloud VPN/Interconnect: eBGP を使用
  Partner Interconnect では Google ASN として 16550 を使用
  Dedicated Interconnect や HA VPN（Cloud Router）では Cloud Router 用の ASN（16-bit 64512–65534 または 32-bit 4200000000–4294967294: RFC 6996）を使用
  お客様の ASN はパブリック・プライベートいずれも利用可能
```

### 13.2 BGPの重要な属性（PCNE試験頻出）

```text
BGPパス属性 = BGPがルートを選択する基準

① AS_PATH（ASパス）:
  そのルートが通過したAS番号のリスト
  短いAS_PATHが優先される（AS数が少ない = より効率的な経路）

  ルートA: 65001 → 65002 → 65003 (3AS経由)
  ルートB: 65001 → 65002 (2AS経由) ← こちらを優先

② LOCAL_PREF（ローカルプリファレンス）:
  同じASN内でどのルートを優先するかの数値
  高い値が優先される
  eBGPで受信したルートにローカルで設定

  Primary Interconnect: LOCAL_PREF = 200 (優先)
  Backup VPN Tunnel:   LOCAL_PREF = 100 (バックアップ)
  → PRIMARY障害時にVPNにフェイルオーバー

③ MED（Multi-Exit Discriminator）:
  隣接ASに対してどの経路を好むかを伝える
  低い値が優先される
  GCPでの用途: フェイルオーバーの方向を制御

BGPルート選択の優先順位（簡略版）:

  1. 最高の LOCAL_PREF
  2. 最短の AS_PATH
  3. 最低の MED
  4. eBGP > iBGP
  5. 最低のルーターID

```

### 13.3 Cloud Router での BGP 設定

```text
Cloud Router の BGP 設定例:

  オンプレ（ASN: 65002）
  └── Interconnect / VPN
       └── Cloud Router（ASN: 65001）
            └── VPC

BGPセッション確立:
  Cloud Router ASN: 65001（Googleが推奨するPrivate ASN）
  オンプレASN:      65002（お客様が設定）

  ルート広告（GCP → オンプレ）:
    VPCサブネット 10.0.0.0/24 を広告
    VPCサブネット 10.0.1.0/24 を広告

  ルート受信（オンプレ → GCP）:
    オンプレ 192.168.0.0/16 を受信してVPCのルートテーブルに追加

フェイルオーバーの設定例:
  # プライマリ: Interconnect (LOCAL_PREF: 200)
  # バックアップ: HA VPN (LOCAL_PREF: 100)

  Interconnect が落ちると:
    → BGPでルートが消える
    → 自動的に VPN のルート（LOCAL_PREF: 100）に切り替わり
    → ダウンタイム: BGPのhold time（最短30秒程度）

BFD（Bidirectional Forwarding Detection）:
  BGPより高速な障害検知プロトコル
  設定すると数秒以内にフェイルオーバーが可能
  Cloud Router + Interconnect で対応
```

### ✅ ベストプラクティス: BGP

```text

1. HA VPN/Interconnect では必ず BGP（動的ルーティング）を使用
2. LOCAL_PREF でプライマリ/バックアップを明確に定義
3. BFDを有効化して高速フェイルオーバーを実現
4. BGPセッションの状態を Cloud Monitoring で監視
5. カスタムルート広告で不要なルートの漏洩を防ぐ
6. ルート受信数に上限を設定（BGPハイジャック対策）

```

> 📎 参照: https://cloud.google.com/network-connectivity/docs/router/concepts/overview
> 📎 参照: https://cloud.google.com/network-connectivity/docs/router/concepts/bfd

## 第3章-14: 高可用性と冗長性設計

### 14.1 可用性の定義と計算

```text
可用性（Availability）= システムが正常に動作している時間の割合

計算式:
  可用性 = 稼働時間 / (稼働時間 + 停止時間) × 100%

SLA（Service Level Agreement）:
  サービス提供者が保証する可用性の数値

  99%    → 年間 87.6時間の停止が許容される
  99.9%  → 年間 8.76時間（= 525.6分）の停止が許容される
  99.99% → 年間 52.56分の停止が許容される
  99.999% → 年間 5.256分の停止が許容される（Five Nines）

計算方法:
  1年間の総時間: 365 × 24 = 8760時間
  99.9%の停止時間: 8760 × 0.001 = 8.76時間 = 525.6分

GCPのSLA（参考）:
  HA VPN:             99.99%（年間 52.56分以内の停止）
  Classic VPN:        99.9% （年間 8.76時間以内の停止）
  Dedicated Interconnect (HA構成): 99.99%
  Cloud Spanner:      99.999%（Five Nines！）
  Compute Engine（単一ゾーン）: 99.99%
  GKE Autopilot:      99.95%
```

### 14.2 障害設計の考え方

```text
単一障害点（SPOF: Single Point of Failure）の排除:

  SPOFあり（危険）:
  [ユーザー] → [ロードバランサー1台] → [サーバー3台]
                    ↑ ここが壊れると全サービス停止！

  SPOFなし（冗長化）:
  [ユーザー] → [グローバルLB（Anycast・自動冗長）] → [サーバー群]
                    複数のGFEで冗長化されている

冗長性の設計パターン:

  Active-Active（アクティブ-アクティブ）:
    2台（以上）が同時に稼働してトラフィックを分担
    1台が障害でも残りで継続
    例: マルチリージョンGKEクラスター

  Active-Passive（アクティブ-スタンバイ）:
    1台が稼働、1台は待機（普段は使わない）
    障害時にPassiveがActiveに切り替わる
    例: Cloud SQL HA構成（リージョナル）

  N+1 冗長:
    N台で必要なスペックを満たし、+1台を予備に
    例: 3台で処理可能なシステムを4台で運用

地理的冗長（ゾーン・リージョン冗長）:
  ゾーン冗長:
    us-central1-a と us-central1-b に各2台
    → 1ゾーンが障害でも継続（同じリージョン内）
    → レイテンシ: 最低

  リージョン冗長:
    us-central1 と europe-west1 に配置
    → リージョン全体の障害（大規模災害等）に対応
    → レイテンシ: 大きい（データ同期コストも高い）

  マルチリージョン:
    GCSやCloud Spannerが対応
    → 複数リージョンに自動レプリケーション
    → 最高の可用性・最も高コスト
```

### 14.3 RTO・RPO の概念

```text
障害復旧の指標（試験頻出）:

RTO（Recovery Time Objective）= 目標復旧時間
  障害発生からシステムが復旧するまでの最大許容時間
  「障害から何分以内に復旧しないといけないか」

  RTO = 0秒: 無停止（Active-Active構成で自動切り替え）
  RTO = 数分: 自動フェイルオーバー（Cloud SQL HA等）
  RTO = 数時間: 手動でのバックアップからの復旧

RPO（Recovery Point Objective）= 目標復旧時点
  障害発生時に許容できるデータ損失量（時間で表現）
  「直近の何分前のデータが失われても許容できるか」

  RPO = 0秒: データ損失なし（同期レプリケーション）
  RPO = 5分: 最大5分前のデータは失われる（非同期レプリケーション）
  RPO = 24時間: 前日のバックアップからの復旧

RTO/RPOと設計の関係:
  RTO低い・RPO低い = コスト最大（リアルタイムレプリケーション必要）
  RTO高い・RPO高い = コスト最小（定期バックアップのみ）

  例:
  ミッションクリティカルな金融システム:
    RTO: 1分以内, RPO: 0秒
    → Cloud Spanner（グローバル分散DB）+ マルチリージョン

  一般的なWebサービス:
    RTO: 15分以内, RPO: 5分
    → Cloud SQL HA + 非同期レプリカ

  社内の分析システム:
    RTO: 4時間, RPO: 24時間
    → 単一ゾーン + 日次バックアップ
```

### ✅ ベストプラクティス: 高可用性設計

```text
可用性を上げるための設計原則:

1. SPOFを排除する

   → 全てのコンポーネントを冗長化
   → 特に「1台しかない」コンポーネントを見つける

2. 障害ドメインを意識する

   → 物理的に独立した場所に分散（ゾーン・リージョン）
   → 同じラック・電源・ネットワークに集中させない

3. ヘルスチェックと自動切り替え

   → ヘルスチェックで障害を自動検知
   → MIG（マネージドインスタンスグループ）で自動復旧

4. 段階的な障害を想定する

   → 「1台障害」だけでなく「1ゾーン全滅」「1リージョン全滅」も想定
   → 定期的なGameDay（障害テスト）実施

5. バックアップと復旧テスト

   → バックアップを取るだけでなく、復旧できることを定期的に確認
   → 「バックアップは取っているが復旧したことがない」は危険
```

> 📎 参照: https://cloud.google.com/architecture/framework/reliability

## 第4章-15: SDN（ソフトウェア定義ネットワーク）

### 15.1 従来のネットワークとSDNの違い

```text
従来のネットワーク（分散型制御）:

  ┌── ルーター A ──┐  ┌── ルーター B ──┐  ┌── ルーター C ──┐
  │  制御プレーン  │  │  制御プレーン  │  │  制御プレーン  │
  │  データプレーン│  │  データプレーン│  │  データプレーン│
  └───────────────┘  └───────────────┘  └───────────────┘
  各ルーターが独自に判断・設定
  → 変更が遅い・設定ミスが発生しやすい

SDN（集中型制御）:

  ┌──────── SDN コントローラー（集中制御）─────────┐
  │  制御プレーン（集中）                         │
  └───────────────┬──────────────────────────────┘
                  │ OpenFlow / REST API
  ┌── スイッチ A ─┴── スイッチ B ─── スイッチ C ──┐
  │  データプレーンのみ（パケット転送に専念）       │
  └───────────────────────────────────────────────┘
  → コントローラーが全スイッチを一括管理
  → 設定変更がAPIで瞬時に反映
```

### 15.2 GCPのSDN（Andromeda）

```text
Google の VPC は「Andromeda」というSDNプラットフォームで実装

Andromeda の特徴:

  - ソフトウェアで実装された仮想ネットワーク
  - 物理ハードウェアなしにネットワーク機能を提供
  - APIで即座に設定変更が可能
  - Google のデータセンター全体を統一的に管理

ユーザーから見た利点（SDNの恩恵）:
  ✓ VPC の作成: API 1回で数秒以内に完成
  ✓ ファイアウォールルールの変更: 即座に全VMに反映
  ✓ ルーティングの変更: ほぼリアルタイムで反映
  ✓ サブネットの追加: ダウンタイムなし
  ✓ グローバルな一貫したネットワーク: 全リージョンが同一VPC

物理ネットワークとの対応:
  物理スイッチ・ルーター → Andromeda（ソフトウェア実装）
  物理ケーブル → Googleのグローバルバックボーン
  VLAN → VPC サブネット
  VRF(仮想ルーティング) → 複数VPC間の分離
```

### 15.3 NFV（Network Functions Virtualization）

```text
NFV = ネットワーク機能を専用ハードウェアからソフトウェアへ

従来:
  物理ファイアウォールアプライアンス（高価・固定容量）
  物理ロードバランサーアプライアンス
  物理VPNゲートウェイ

NFV後:
  ソフトウェアFW（クラウドFW）→ スケール自在・低コスト
  ソフトウェアLB（Cloud LB） → API で瞬時に設定変更
  ソフトウェアVPN（Cloud VPN）→ 即座にプロビジョニング

GCPでのNFV:
  Cloud Armor = 仮想WAF
  Cloud Load Balancing = 仮想ロードバランサー
  Cloud VPN = 仮想VPNゲートウェイ
  Cloud NAT = 仮想NATゲートウェイ
  Cloud DNS = 仮想DNSサーバー
  → 全てソフトウェアで実装 = NFV の実践
```

> 📎 参照: https://cloud.google.com/vpc/docs/vpc#vpc_networks_and_subnets

## 第4章-16: CDNとエッジネットワーク

### 16.1 CDN（Content Delivery Network）の基本原理

```text
CDNが解決する問題:

  CDNなし（オリジンサーバーから直接配信）:

  東京のユーザー → [太平洋を横断] → ニューヨークのサーバー
  → レイテンシ: 200〜300ms（遠すぎる）
  → サーバー負荷: 全世界のアクセスが1か所に集中

  CDNあり（エッジキャッシュから配信）:

  東京のユーザー → [東京のEdgeサーバー（キャッシュ）]
                      → ミリ秒で応答（物理的に近い）

  東京に無い場合のみ:
  東京Edge → [ニューヨークのオリジンサーバー] → キャッシュ保存

CDNの仕組み（詳細）:

  1. ユーザーがコンテンツをリクエスト
  2. DNSがAnycasting/ジオ解決で最寄りEdgeにルーティング
  3. EdgeサーバーがキャッシュをチェックHIT/MISS判断

     ・CACHE HIT: Edgeから即座に応答（コンテンツを返す）
     ・CACHE MISS: オリジンからコンテンツを取得してキャッシュに保存

  4. TTL（生存時間）が切れたら再検証・更新

```

### 16.2 キャッシュの仕組みと制御

```text
HTTPキャッシュヘッダー:

Cache-Control: public, max-age=86400
  public    → CDN含む全キャッシュで保存可
  private   → ブラウザのみ（CDNはキャッシュしない）
  no-cache  → 毎回オリジンで再検証（変更を素早く反映したい時）
  no-store  → キャッシュ禁止（機密情報・認証後のコンテンツ）
  max-age   → 秒単位でキャッシュ有効期間を指定

ETagとCacheの連携:
  オリジンがコンテンツにETAGを付与（例: "abc123"）
  CDNがキャッシュと一緒にETAGを保存
  TTL後、CDNがオリジンに "etag: abc123 で変更あり?" と問い合わせ
  → 変更なし (304 Not Modified) → キャッシュをそのまま使用
  → 変更あり (200 OK) → 新しいコンテンツを取得

Cloud CDNのキャッシュモード:
  CACHE_ALL_STATIC:
    静的コンテンツ（画像・CSS・JS等）を自動キャッシュ
    デフォルト設定

  USE_ORIGIN_HEADERS:
    オリジンの Cache-Control ヘッダーに完全に従う
    細かい制御が必要な場合

  FORCE_CACHE_ALL:
    全レスポンスをキャッシュ
    オリジンがキャッシュ禁止ヘッダーを設定していても無視

キャッシュキー（何でキャッシュを識別するか）:
  デフォルト: URL全体（プロトコル + ホスト + パス + クエリ）
  カスタム: 特定のヘッダー・クエリパラメータを除外/追加

  例: www.example.com/image.jpg?quality=high&user=123
  → user=123 をキャッシュキーから除外すると
  → www.example.com/image.jpg?quality=high で統一キャッシュ
  → 同じ画像をユーザーごとに別キャッシュせずに済む（効率化）
```

### ✅ ベストプラクティス: CDN設計

```text

1. 静的コンテンツ（画像・動画・JS・CSS）は積極的にキャッシュ
2. 動的コンテンツ（API・ユーザー固有）はno-cacheまたはprivate
3. 機密情報（認証後コンテンツ）はno-storeで確実にキャッシュ無効化
4. キャッシュヒット率を監視（80%以上を目指す）
5. キャッシュの無効化（Invalidation）を計画的に使用
6. Signed URLでプレミアムコンテンツを保護

```

> 📎 参照: https://cloud.google.com/cdn/docs/caching

## 第4章-17: ハイブリッドクラウドネットワーク

### 17.1 ハイブリッドネットワークとは？

```text
ハイブリッドネットワーク = オンプレミスとクラウドを組み合わせた環境

なぜハイブリッドが必要か:
  ① レガシーシステム: 移行困難な古いシステムがオンプレに残る
  ② 規制対応: 特定のデータを自社施設内に保管する義務
  ③ 段階的移行: いきなりフルクラウドに移行できない
  ④ レイテンシ: オンプレとクラウドの連携が必要なシステム
  ⑤ コスト: 既存の設備投資を活かす

ハイブリッド接続の選択肢（帯域とコストのトレードオフ）:

  ┌─────────────────────────────────────────────────────────┐
  │                         コスト              │
  │        低  ─────────────────────────→  高  │
  │ 帯域 低 Cloud VPN (IPsec)                              │
  │          ・最大3Gbps/トンネル                          │
  │          ・インターネット経由（暗号化）                  │
  │          ・最も低コスト・即日開通                        │
  │                                                        │
  │      中  Partner Interconnect                          │
  │          ・50Mbps〜50Gbps                              │
  │          ・専用線（パートナー経由）                      │
  │          ・中コスト・1〜4週間で開通                      │
  │                                                        │
  │ 帯域 高  Dedicated Interconnect                        │
  │          ・10Gbps または 100Gbps                       │
  │          ・Googleと直接の専用線                         │
  │          ・高コスト・数週間〜数ヶ月で開通                │
  └─────────────────────────────────────────────────────────┘
```

### 17.2 ハイブリッド接続の選択基準

```text
選択フローチャート:

  帯域要件は何 Gbps 以上か?
  │
  ├── 3 Gbps 未満かつインターネット品質で許容できる?
  │   → Cloud VPN（最もシンプル・低コスト）
  │
  ├── 3 Gbps 以上かつ Google のコロケーションに接続できる？
  │   → Dedicated Interconnect（最高帯域・最低レイテンシ）
  │
  └── それ以外（または段階的な帯域増加が必要）?
      → Partner Interconnect（柔軟な帯域・接続場所）

可用性要件:
  99.9% → Cloud VPN Classic または Interconnect単線
  99.99% → HA VPN（2トンネル）または Interconnect 2Metro・4回線

コスト最適化:
  小規模テスト・開発環境: Cloud VPN
  本番・ミッションクリティカル: Dedicated Interconnect + HA VPN（バックアップ）
  月額帯域が変動する: Partner Interconnect（従量課金可能）
```

### 17.3 マルチクラウドネットワーク

```text
マルチクラウド = 複数のクラウドプロバイダーを組み合わせる

なぜマルチクラウドか:
  ① ベンダーロックイン回避
  ② 各クラウドの得意分野を活用
  ③ 障害時の冗長性
  ④ 規制（特定クラウドの使用義務）

GCP ↔ AWS・Azure 間の接続方法:
  ① インターネット経由（最もシンプル・暗号化必須）
  ② VPN（GCP Cloud VPN ↔ AWS VPN Gateway）
  ③ Cross-Cloud Interconnect（専用線・新機能）

GCP の Cross-Cloud Interconnect:
  Google Cloud と AWS/Azure を専用線で接続
  インターネットを経由しない高品質・低レイテンシ接続
  帯域: 10Gbps または 100Gbps

  ユースケース:
  → GCPのAI/ML処理 ↔ AWSのデータウェアハウス
  → GCPのKubernetes ↔ Azureの既存システム
  → 段階的なクラウド移行（一部のみGCPに移行）

Network Connectivity Center (NCC) でのマルチクラウド:
  NCC Hub: Googleネットワーク内
  Spoke A: オンプレミス（Interconnect経由）
  Spoke B: AWS（Cross-Cloud Interconnect経由）
  Spoke C: Azure（Cross-Cloud Interconnect経由）
  → 全拠点がNCC Hub経由でシームレスに通信
```

### ✅ ベストプラクティス: ハイブリッド設計

```text

1. 接続方法は要件（帯域・コスト・可用性）に基づいて選択
2. 本番環境はHA VPN + Interconnect の組み合わせで冗長化
3. BGPで動的ルーティング（手動管理のコスト削減）
4. 帯域監視を設定（使用率80%でアラート・増強計画を開始）
5. セキュリティ: 全通信を暗号化（VPN or 専用線でも暗号化を推奨）
6. 定期的な接続テスト（本番障害前に手順を確認）

ハイブリッド移行のガイドライン:
  Phase 1: ネットワーク接続確立（VPN/Interconnect）
  Phase 2: DNS の統合（転送ゾーン設定）
  Phase 3: 低リスクワークロードから移行
  Phase 4: データ同期・レプリケーション設定
  Phase 5: トラフィックの段階的な切り替え
  Phase 6: 旧システムの廃止
```

## 📚 総まとめ：PCNE 試験前の重要ポイント集

## 必ず覚えるべき10の核心知識

```text

1. OSI モデルとレイヤー別GCPサービスの対応

   L7 = HTTP(S) LB・Cloud Armor・CDN・DNS
   L4 = Network LB・TCP/SSL Proxy LB
   L3 = Cloud Router・ファイアウォール・Cloud NAT

2. CIDR とサブネット計算

   /24 = 254ホスト、/28 = 14ホスト
   GCPは最初の4つと最後の1つを予約

3. BGP の重要属性

   LOCAL_PREF: 高い値優先（フェイルオーバー制御）
   AS_PATH: 短いほど優先
   MED: 低い値優先（隣接ASへのヒント）

4. TCPの3ウェイハンドシェイク

   SYN → SYN-ACK → ACK（ステートフルFWはこれを追跡）

5. DNS レコードタイプ

   A: ドメイン→IPv4、AAAA: ドメイン→IPv6
   CNAME: エイリアス、MX: メール、TXT: テキスト

6. NAT の種類とCloud NATの特徴

   Cloud NAT = PAT（ポートアドレス変換）
   アウトバウンドのみ（インバウンドは不可）

7. VPN と Interconnect の SLA と帯域

   HA VPN: 99.99% 最大3Gbps/トンネル
   Dedicated Interconnect: 10G/100G

8. TLS バージョンとSSLポリシー

   TLS 1.0/1.1 = 廃止推奨
   TLS 1.2 = 現在の標準（Modern Policy）
   TLS 1.3 = 最新・最高速

9. 高可用性の設計原則

   SPOF排除・ゾーン分散・RTO/RPO の定義

10. ゼロトラストとIAPの概念

    「境界型」から「常時認証」へ
    IAP = VPNなしのセキュアアクセス
```

## よく混同される概念の整理

| 混同パターン                  | 正しい理解                                          |
| ----------------------------- | --------------------------------------------------- |
| L4 LB でURLルーティング       | URLルーティングはL7のみ（HTTP(S) LB）               |
| Cloud NAT でインバウンド      | Cloud NATはアウトバウンドのみ                       |
| BGP LOCAL_PREFは低い値優先    | LOCAL_PREFは高い値優先（MED は低い値優先）          |
| /24 = 256ホスト               | 256 - 2（NW+BC）= 254、GCPはさらに-3                |
| TCP は接続確立なし            | 接続確立なし = UDP                                  |
| VPN で物理回線同等            | 物理的品質 = Interconnect、VPN はインターネット経由 |
| HA VPN = 1本のトンネル        | HA VPN = 2本のトンネルで冗長化                      |
| DNS A レコードで CNAME と同じ | Zone Apex（ドメイン自体）にCNAMEは設定不可          |
