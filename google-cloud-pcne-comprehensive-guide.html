1.1 VPCの根本概念 ─ グローバルスコープとモード選択 ▼

Google Cloud VPCはグローバルリソースです。他のクラウド（AWSなど）ではVPCがリージョンに紐づきますが、GCPでは1つのVPCが複数リージョンにまたがります。これにより、リージョン間のルーティングが自動化され、アーキテクチャが大幅に簡素化されます。

┌──────────── 単一グローバルVPC (example-vpc) ──────────────────────┐ │ │ │ us-central1 europe-west1 asia-northeast1 │ │ subnet: 10.0.1.0/24 subnet: 10.0.2.0/24 subnet: 10.0.3.0/24 │ │ │ │ ↕ Googleのプレミアムバックボーンで自動接続 ↕ │ │ │ └────────────────────────────────────────────────────────────────────┘ ▶ サブネットはリージョンスコープ（ゾーンをまたぐ） ▶ VPC自体はグローバルスコープ（リージョンをまたぐ）

Auto Mode vs Custom Mode

モード サブネット作成 CIDRレンジ 推奨用途 本番環境
Auto Mode 全リージョンに自動作成 10.128.0.0/9（固定） 学習・プロトタイプ・PoC ❌ 非推奨
Custom Mode 手動で設計・作成 任意のRFC1918レンジ 本番・エンタープライズ ✓ 推奨
⚠
Auto Modeを本番禁止の理由： 10.128.0.0/9が固定されるため、オンプレミスのCIDRと重複しやすく、ハイブリッド接続を後から追加するときに大きな問題が発生します。最初からCustom Modeを使うのが鉄則です。

CIDR設計のベストプラクティス

# RFC 1918 プライベートIPレンジの使い分け 10.0.0.0/8 → 大規模組織（16,777,214ホスト） ← 最もよく使用 172.16.0.0/12 → 中規模組織（1,048,574ホスト） 192.168.0.0/16 → 小規模組織（65,534ホスト） # 環境ごとのVPC設計例（将来の拡張を見越して余裕を持たせる） Production VPC: us-central1 → 10.1.0.0/16 (65,534ホスト) web-subnet → 10.1.1.0/24 (254ホスト) app-subnet → 10.1.2.0/24 (254ホスト) db-subnet → 10.1.3.0/24 (254ホスト) europe-west1 → 10.2.0.0/16 asia-northeast1 → 10.3.0.0/16 Staging VPC: 10.10.0.0/16 Development VPC: 10.20.0.0/16

✅ ベストプラクティス

本番環境では必ずCustom Mode VPCを使用する（Auto Modeは絶対禁止）
環境ごとにVPCを分離する（prod / staging / dev で別VPC）
オンプレミスCIDRとの重複を設計初期から排除する
将来のリージョン拡張・サブネット追加を見越してCIDRに余裕を持たせる
GKE利用時はセカンダリレンジ（Pod/Service用）を事前計画する

1.2 VPCファイアウォールルール ─ ステートフル・優先度・階層型ポリシー ▼

GCPのファイアウォールはステートフルです。インバウンドを許可すれば戻りトラフィックは自動的に許可されます。また、ルールはVMの外側（vNICレベル）で適用されるため、OSのiptablesとは独立して動作します。

ターゲット指定の3つの方法

🏷
方法1: ネットワークタグ
VMにタグ（例: web-server）を付与し、そのタグを持つVMにルールを適用。シンプルだが誰でもタグを付けられるためセキュリティは低め。
🔐
方法2: サービスアカウント（推奨）
VMに割り当てたService Accountでルールを制御。SA付与にはIAM権限が必要なため、タグより安全。エンタープライズ環境では優先使用。
🌐
方法3: ネットワーク全体
ターゲットを指定しない場合はVPC内の全VMに適用される。細粒度制御には不向き。基本的なベースラインルールにのみ使用。

ルール優先度と評価順序

# 優先度: 数値が小さいほど高優先度 (範囲: 0〜65535) Priority 100: DENY → src: 10.0.0.5/32, port: 22 ← 先に評価される Priority 200: ALLOW → src: 10.0.0.0/24, port: 22 Result: 10.0.0.5 からのSSH → DENY (Priority 100 がマッチ) 10.0.0.1 からのSSH → ALLOW (Priority 100 はマッチせず、Priority 200 がマッチ) 10.9.9.9 からのSSH → DENY (暗黙のDENY: priority 65535) # デフォルトの暗黙ルール（変更不可） - 全Egress（外向き）ALLOW priority: 65535 - 全Ingress（内向き）DENY priority: 65535

階層型ファイアウォールポリシー（Hierarchical Firewall Policies）

組織レベルでファイアウォールを一元管理できる強力な機能。評価順序は 組織 → フォルダ → プロジェクト（VPCルール） の順です。

# 評価順序（上から下へ） [組織レベルポリシー] Rule: DENY malicious IPs → "DENY"ならここで終了 Rule: GOTO_NEXT → 下位に評価を委譲 ↓ [フォルダレベルポリシー（部門別）] Rule: ALLOW internal tools → 適用 Rule: GOTO_NEXT ↓ [プロジェクト VPCファイアウォールルール] Rule: ALLOW HTTP/HTTPS → 適用
💡
試験ポイント： GOTO_NEXT アクションを設定した場合のみ下位に評価が移ります。上位で ALLOW または DENY が決まれば、下位のルールは評価されません。

サービスアカウントベースのFWルール例

# アプリ層→DB層への通信のみ許可（SAベース） gcloud compute firewall-rules create allow-app-to-db \ --direction=INGRESS \ --network=my-vpc \ --action=ALLOW \ --rules=tcp:5432 \ --source-service-accounts=app-sa@project.iam.gserviceaccount.com \ --target-service-accounts=db-sa@project.iam.gserviceaccount.com

✅ ベストプラクティス

最小権限の原則: 必要なポートのみ開放、デフォルト拒否を維持
タグよりもサービスアカウントベースのルールを優先する
全ルールでファイアウォールログを有効化する（セキュリティ監査・トラブルシュート）
階層型ポリシーで組織全体のベースラインを設定し、プロジェクトルールで細分化する
SSHはIAP（35.235.240.0/20）経由のみ許可し、0.0.0.0/0は絶対に開けない
ヘルスチェック用IP範囲（35.191.0.0/16, 130.211.0.0/22）を必ず許可する

1.3 VPCピアリング vs Shared VPC ─ 設計パターンの選択 ▼

VPCピアリング（VPC Network Peering）

異なるVPC間でプライベートIPを使って直接通信する仕組み。両方向に設定が必要です。

🚫
推移的ルーティングは不可： A↔B、B↔Cとピアリングしても、AとCは通信できません。A-C間にも直接ピアリングが必要です。試験で最も多く問われる制約です。
# 推移的ルーティング（これはできない） VPC-A ──ピアリング── VPC-B ──ピアリング── VPC-C A→B: OK | B→C: OK | A→C: NG ❌ # 解決策：A-C間にも直接ピアリングを設定 VPC-A ──ピアリング── VPC-B └────────ピアリング────────→ VPC-C

Shared VPC（共有VPC）

ホストプロジェクトのVPCを複数のサービスプロジェクトで共有する仕組み。ネットワーク管理を中央集権化できます。

┌── Host Project（ネットワーク専用プロジェクト）──────────────────┐ │ VPC: company-vpc / サブネット・FW・VPN・Interconnect を管理 │ └──────────────────────────────────────────────────────────────────┘ ↕ Shared VPC ↕ Shared VPC ┌── Service Project A ─────┐ ┌── Service Project B ─────┐ │ チームA のVMをデプロイ │ │ チームB のVMをデプロイ │ │ Host VPCのサブネット使用 │ │ Host VPCのサブネット使用 │ └──────────────────────────┘ └──────────────────────────┘

どちらを使うべきか？

観点 VPC Peering Shared VPC
対象組織 異なる組織間・パートナー連携 同一組織内の複数プロジェクト
ネットワーク管理 各VPCが独立して管理 ホストプロジェクトが一元管理
推移的ルーティング 不可 可能（同一VPC内）
CIDR重複 重複不可（設計注意） 共有VPCなので問題なし
推奨場面 異組織間接続 企業内の部門間分離（推奨）

✅ ベストプラクティス

大企業・マルチプロジェクト環境ではShared VPCを標準として採用する
Host ProjectはVPC専用プロジェクトとして独立させる（他のリソースは置かない）
サービスプロジェクトへの権限はサブネット単位でroles/compute.networkUserを付与
組織ポリシーでShared VPC外のVPC作成を制限してガバナンスを確保
ピアリング設計時はCIDR重複を事前に徹底チェックする

1.4 Cloud NAT・Private Google Access・Private Service Connect ▼

Cloud NAT

外部IPを持たないVMがインターネットへアウトバウンド通信できるようにするマネージドNATサービス。インバウンド接続は一切不可です。

⚠
試験頻出： Cloud NATはアウトバウンドのみです。「外部からVMへの接続」に使えると思い込む誤りが多発します。受信接続が必要な場合はロードバランサーか外部IPを使います。
# 1. Cloud Routerを作成（NAT必須の依存関係） gcloud compute routers create my-router \ --network=my-vpc \ --region=us-central1 # 2. Cloud NAT Gatewayを作成 gcloud compute routers nats create my-nat \ --router=my-router \ --auto-allocate-nat-external-ips \ --nat-all-subnet-ip-ranges \ --region=us-central1

Private Google Access

外部IPを持たないVMが、Googleサービス（GCS・BigQuery等）へアクセスできる機能。サブネット設定で有効化するだけです。

通常: VM(外部IPなし) → インターネット経由 → Google Public API ↑ 外部IPがないと到達不可！ PGA有効時: VM(外部IPなし) → 199.36.153.8/30 → Google内部ネットワーク → GCS/BQ # 設定コマンド（サブネットで有効化） gcloud compute networks subnets update my-subnet \ --enable-private-ip-google-access \ --region=us-central1

Private Service Connect（PSC）

マネージドサービス（Cloud SQL・GCS等）へVPCピアリングなしでプライベートアクセスできる機能。シンプルなエンドポイント経由で接続します。

機能 Private Google Access Private Service Connect
用途 Google標準API（GCS・BQ等）へのアクセス マネージドサービス・サードパーティへの接続
設定 サブネット設定で有効化のみ ForwardingRule + ServiceAttachmentが必要
IP管理 自社VPC内からGoogleのAPI IPへ 自社VPC内の任意IPでサービスに接続
VPCピアリング 不要 不要（PSC自体がピアリングを代替）

✅ ベストプラクティス

全サブネットでPrivate Google Accessを有効化する（外部IPなしVMのGCPサービスアクセス確保）
Dynamic Port Allocationを使用してNATポートを効率的に利用する
本番環境では予約外部IPをCloud NATに割り当てる（IP変動を防ぐ）
マネージドサービスへのアクセスはPSCを優先（VPC Peeringより管理コスト低）
NATログを有効化してポート枯渇を監視する

2.1 接続方式の全体比較 ─ 帯域・コスト・SLAで選択する ▼
接続方式 帯域幅 経路 SLA 暗号化 コスト 推奨場面
HA VPN 最大 3Gbps/トンネル インターネット経由 99.99% IPsec（必須） 低コスト 帯域が少なく、コスト優先の場合
Classic VPN 最大 3Gbps/トンネル インターネット経由 99.9% IPsec（必須） 低コスト 新規非推奨

① 大帯域（数十Gbps以上）+ 低レイテンシ必須 → Dedicated Interconnect
② Googleコロケ施設に物理接続できない → Partner Interconnect
③ コスト優先・帯域が3Gbps以下で十分 → HA VPN
④ Classic VPNは新規構築で使わないこと（99.9% SLAのみ）
2.2 HA VPN ─ 99.99% SLAを実現する高可用性VPN設計 ▼

HA VPN（High Availability VPN）は99.99% SLAを提供します。2つの独立したインターフェースに2本のトンネルを張り、BGPで動的ルーティングを行います。

オンプレ VPN GW Google HA VPN GW ┌──────────────────┐ ┌──────────────────┐ │ Interface 0 │──tunnel1──│ Interface 0 │ ← 独立した外部IP │ (IP: x.x.x.x) │ │ (IP: a.a.a.a) │ │ │ │ │ │ Interface 1 │──tunnel2──│ Interface 1 │ ← 独立した外部IP │ (IP: y.y.y.y) │ │ (IP: b.b.b.b) │ └──────────────────┘ └──────────────────┘ → どちらかのトンネルが落ちても自動フェイルオーバー → BGP（Cloud Router）がルートを自動更新
# Step1: HA VPN Gatewayを作成 gcloud compute vpn-gateways create my-ha-vpn-gw \ --network=my-vpc \ --region=us-central1 # Step2: Cloud Routerを作成（BGP用） gcloud compute routers create my-vpn-router \ --network=my-vpc \ --region=us-central1 \ --asn=65001 # Step3: 外部VPN GWを定義（オンプレ側） gcloud compute external-vpn-gateways create on-prem-gw \ --interfaces 0=ON_PREM_IP_0,1=ON_PREM_IP_1 # Step4: トンネル2本を作成（冗長化必須） gcloud compute vpn-tunnels create tunnel-1 \ --vpn-gateway=my-ha-vpn-gw \ --interface=0 \ --peer-external-gateway=on-prem-gw \ --peer-external-gateway-interface=0 \ --shared-secret=MY_STRONG_SECRET \ --ike-version=2 \ --router=my-vpn-router \ --region=us-central1 # tunnel-2も同様に作成（--interface=1）

✅ ベストプラクティス

新規VPN構築は必ずHA VPNを使用（Classic VPNは非推奨）
IKEv2を使用（IKEv1より安全で効率的）
Pre-Shared Keyは強力なものを使用（gcloud自動生成推奨）
BGP（動的ルーティング）を必ず設定する（静的ルートは管理コストが高い）
帯域不足の場合はECMP（等コストマルチパス）でトンネルをスケールする

2.3 Cloud Interconnect ─ 専用線による99.99%冗長設計 ▼

Dedicated InterconnectはGoogleのコロケーション施設に物理的に直接接続する専用線サービスです。10Gbps/100Gbps単位で回線を確保します。

お客様データセンター ┌──────────────────────┐ │ お客様ルーター │ │ (BGP: ASN 65002) │ └──────────┬───────────┘ │ 専用物理回線（10G/100G） ┌──────────┴───────────┐ │ Google Meet-Me │ ← Googleが指定するコロケーション施設 │ Location (Equinix等) │ └──────────┬───────────┘ │ Google内部ネットワーク ┌──────────┴───────────┐ │ Google Cloud VPC │ │ (VLAN Attachment経由) │ └──────────────────────┘

99.99% SLA冗長構成（本番環境必須）

Metro-A（物理的拠点A） Metro-B（物理的拠点B） ┌────────────────────┐ ┌────────────────────┐ │ Interconnect 1 (10G)│ │ Interconnect 3 (10G)│ │ Interconnect 2 (10G)│ │ Interconnect 4 (10G)│ └────────────────────┘ └────────────────────┘ ↕ VLAN Attachment ↕ VLAN Attachment ↕ Cloud Router (BGP) ↕ Cloud Router (BGP) ↕ Cloud VPC ↕ Cloud VPC → Metro-A が完全停止してもMetro-Bで継続 = 99.99% SLA → 同一Metro内2本のみ = 99.9% SLA（非推奨）

VLAN Attachmentとは

1本の物理専用線を論理的に複数に分割する仕組み（VLAN）。各アタッチメントが1つのVPCに対応し、1本の回線で複数のVPCに接続できます。

✅ ベストプラクティス

本番環境は99.99% SLA構成（異なる2 Metro × 2回線 = 合計4回線）
VLAN Attachmentを使ってVPCごとに論理分離する
BGPでルートを動的管理する（Cloud Router必須）
暗号化が必要な場合はHA VPN over Interconnectを使用

混同しやすいポイント ─ 試験の落とし穴
受験者が誤りやすい概念の対比をまとめました。試験直前に必ず確認してください。
❌ 「VPCはリージョンスコープだ」

VPC自体はグローバルスコープです。リージョンスコープなのはサブネットです。1つのVPCが複数リージョンにまたがれるのがGCPの特徴です。

❌ 「Cloud VPN（全般）は99.99% SLA」

HA VPNが99.99% SLAです。Classic VPNは99.9% SLAです。新規構築ではHA VPNを使用してください。

❌ 「Cloud NATでインバウンド接続も受けられる」

Cloud NATはアウトバウンドのみです。外部からの受信接続はできません。受信が必要な場合はロードバランサーや外部IPを使用します。

❌ 「VPC PeeringでA-B-Cがつながれば全VPCが通信できる」

推移的ルーティングは不可です。A→B→CとピアリングしてもAとCは通信できません。A-C間にも直接ピアリングが必要です。

❌ 「Cloud ArmorはすべてのLBで使える」

Cloud ArmorはProxy型LB（Application LB・TCP Proxy LB等）にのみ対応します。Passthrough型（Network LB・Internal TCP/UDP LB）では使えません。

❌ 「Dedicated InterconnectはどこでもGoogleと直接接続できる」

Dedicated InterconnectはGoogleが指定するコロケーション施設（Meet-Me Location）に物理接続できる場合のみ利用可能です。施設に接続できない場合はPartner Interconnectを使用します。

❌ 「IAMポリシーだけでデータ漏洩を完全に防げる」

IAMはユーザーのアクセス権限を制御しますが、正規ユーザーによるデータ持ち出し（Exfiltration）は防げません。VPC Service ControlsでAPIの境界を設定することでデータ漏洩を防止します。

試験当日の解答戦略

🎯
最もシンプルな解決策を選ぶ
複雑な構成より管理が簡単なソリューションが正解になりやすい。「これをしなくてもあれで解決できる」という選択肢を探す。
☁
マネージドサービスを優先
自分で管理するより「Googleが管理するマネージドサービス」を選ぶのが基本方針。運用コストが低く、SLAも高い。
🔄
冗長性・HAの要件を必ず確認
99.99% SLAが必要ならHA VPN・Interconnect複数Metro。SLAの数字と対応する構成を暗記する。
💰
コストと要件のバランス
Dedicated Interconnectが必須かVPNで十分かを見極める。帯域・レイテンシ要件が鍵になる。
🔐
セキュリティは最小権限
最も限定的なアクセスを許可するソリューションを選ぶ。「最小権限の原則」が常に正しい方向を示す。
📡
帯域要件で接続方式を選ぶ
大帯域→Interconnect、中帯域→Partner IC、小帯域/コスト優先→VPN。この3段階の判断軸を常に意識する。
