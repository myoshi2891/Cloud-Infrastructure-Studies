## Chapter 1: Gmail の設定（2.1）

### 1.1 MX レコードによるメールルーティングの設定

#### 🔰 MX レコードとは何か

**MX（Mail Exchanger）レコード**は、そのドメイン宛のメールをどのメールサーバーが受け取るかを DNS に登録するレコードです。
Google Workspace で Gmail を使うためには、このレコードを Google のメールサーバーに向ける必要があります。

```text
【MX レコードの仕組み】

送信者 → DNSで example.com の MX レコードを照会
                    ↓
         "ASPMX.L.GOOGLE.COM が受け取る"
                    ↓
         Google のメールサーバーにメールを送信
                    ↓
         受信者の Gmail 受信箱に届く
```

#### Google Workspace の MX レコード設定値

DNS レジストラ（お名前.com、Route 53、Cloudflare など）に以下を設定します。

**【推奨】新しい Workspace セットアップ（シングルレコード）**
| 優先度（Priority） | メールサーバー（Value）   |
| ------------------ | ------------------------- |
| 1                  | `smtp.google.com`         |

**【レガシー互換】既存のセットアップ向け（マルチレコード）**
| 優先度（Priority） | メールサーバー（Value）   |
| ------------------ | ------------------------- |
| 1                  | `ASPMX.L.GOOGLE.COM`      |
| 5                  | `ALT1.ASPMX.L.GOOGLE.COM` |
| 5                  | `ALT2.ASPMX.L.GOOGLE.COM` |
| 10                 | `ALT3.ASPMX.L.GOOGLE.COM` |
| 10                 | `ALT4.ASPMX.L.GOOGLE.COM` |

> ⚠️ **重要**: MX レコードの設定変更後、DNS の伝播には最大 **48 時間** かかります。設定直後は旧サーバーにメールが届く場合があるため、移行時は注意が必要です。

#### ✅ ベストプラクティス：MX レコード

| #   | ベストプラクティス                                                               | 理由                                 |
| --- | -------------------------------------------------------------------------------- | ------------------------------------ |
| 1   | **移行前は旧 MX レコードを保持したまま Google の MX を追加し、TTL を下げておく** | 切り替え時のダウンタイムを最小化する |
| 2   | **切り替え後は旧 MX レコードを削除する**                                         | セキュリティリスクと設定の混乱を防ぐ |
| 3   | **Admin Toolbox の MX チェッカーで設定を確認する**                               | 設定ミスを早期発見できる             |

> 🔗 **参考**: <https://support.google.com/a/answer/174125>
> 🔗 **Admin Toolbox MX チェッカー**: <https://toolbox.googleapps.com/apps/checkmx/>

### 1.2 基本的なメールルーティングシナリオ

#### メールルーティングの概念図

```text
インターネット
    ↓
[受信ゲートウェイ（オプション）]
    ↓
Google のメールサーバー
    ↓
┌─────────────────────────────────┐
│       Gmail のルーティングエンジン │
│                                 │
│  ① スパム・ウイルスフィルタ      │
│  ② コンテンツコンプライアンス    │
│  ③ ルーティングルール（転送先）  │
│  ④ ユーザーの受信箱へ           │
└─────────────────────────────────┘
```

#### Admin コンソールでの設定場所

```text
Admin コンソール → アプリ → Google Workspace → Gmail → ルーティング
```

#### 主要なルーティング設定の種類

| 設定名                       | 用途                                           | 具体例                                                |
| ---------------------------- | ---------------------------------------------- | ----------------------------------------------------- |
| **デフォルトのルーティング** | 特定のメールを別サーバーに転送                 | `sales@example.com` 宛を別のメールサーバーへ転送      |
| **受信ゲートウェイ**         | サードパーティのスパムフィルタを経由させる     | Proofpoint → Google の順でフィルタリング              |
| **送信ゲートウェイ**         | Google 経由で送信した後、別サーバーにも送信    | アーカイブシステムへの同時送信                        |
| **デュアル配信**             | 同じメールを2つのシステムに同時配信            | 移行期間中、旧システムと新システムの両方に配信        |
| **スプリット配信**           | ユーザーごとに異なるシステムにメールを振り分け | 一部ユーザーは Google、残りは旧システムを使用する場合 |

### 1.3 コンテンツコンプライアンスルール

#### 🔰 コンテンツコンプライアンスとは

**コンテンツコンプライアンスルール**は、メールの本文・件名・添付ファイルの**内容に基づいて**、メールを自動的にフィルタリング・ルーティング・変更する機能です。

```text
【コンテンツコンプライアンスの動作フロー】

メール受信（または送信）
    ↓
コンテンツルールを評価（件名・本文・添付ファイルをスキャン）
    ↓ キーワードや正規表現に一致した場合
アクション実行：
  ├── メールを拒否（バウンスさせる）
  ├── 隔離（管理者レビュー用のキューに入れる）
  ├── 別のメールアドレスに転送（BCCなど）
  ├── ヘッダーを追加・変更
  └── メールを変更（件名の追加、フッターの挿入など）
```

#### コンテンツコンプライアンスルールの設定手順

```text
Admin コンソール → Gmail → コンプライアンス →
コンテンツコンプライアンス → 設定を追加
```

**設定項目**:

| 項目                         | 説明                                                        |
| ---------------------------- | ----------------------------------------------------------- |
| **影響を受けるメール**       | 受信 / 送信 / 組織内 のどれに適用するか                     |
| **コンテンツの追加条件**     | 件名・本文・ヘッダー・添付ファイルを指定                    |
| **条件の種類**               | キーワード一致 / 正規表現 / 添付ファイルタイプ / メタデータ |
| **一致した場合のアクション** | 上記の各アクションを選択                                    |

**使用例（ルールの例）**:

```text
例1：社外秘情報の漏洩防止
  条件: 送信メールの本文に「社外秘」または「CONFIDENTIAL」が含まれる
  かつ: 送信先が組織外のドメイン
  アクション: 隔離（管理者がレビューするまで送信を保留）

例2：競合他社へのメール監視
  条件: 送信先に競合他社ドメイン (@competitor.com) が含まれる
  アクション: コンプライアンス担当者 (bcc:compliance@example.com) に BCC コピー

例3：特定キーワードの自動タグ付け
  条件: 受信メールの件名に「URGENT」が含まれる
  アクション: X-Custom-Header を追加して優先タグを付ける
```

> 🔗 **参考**: <https://support.google.com/a/answer/1346936>

### 1.4 スパム・フィッシング・マルウェア設定

#### 🔰 Gmail のセキュリティ機能の全体像

Gmail は多層的なセキュリティフィルタリングを提供しています。

```text
受信メール
    ↓
【Layer 1】IPレピュテーション評価（送信元IPの信頼度確認）
    ↓
【Layer 2】スパムフィルタ（機械学習による自動分類）
    ↓
【Layer 3】フィッシング対策（偽サイトへのリンク検出）
    ↓
【Layer 4】マルウェアスキャン（添付ファイルのサンドボックス分析）
    ↓
【Layer 5】管理者設定（カスタムルール・許可リスト）
    ↓
ユーザーの受信箱
```

#### 主要なセキュリティ設定一覧

```text
Admin コンソール → Gmail → スパム、フィッシング、マルウェア
```

| 設定名                                  | 機能                                       | 推奨設定                                        |
| --------------------------------------- | ------------------------------------------ | ----------------------------------------------- |
| **スパムフィルタ強化**                  | 迷惑メールの検出強度                       | 「強化されたスパム対策」を有効化                |
| **許可リスト（Email Allowlist）**       | 常に受信トレイに届ける IP アドレスのリスト | 信頼できるビジネスパートナーのメールサーバー IP |
| **ブロックリスト**                      | 特定のメールアドレス・ドメインをブロック   | スパム送信者、不審なドメイン                    |
| **受信ゲートウェイ（Inbound Gateway）** | サードパーティのスパムフィルタを経由させる | セキュリティベンダーのゲートウェイ IP を登録    |
| **IP 許可リスト**                       | 特定 IP からのメールをスパムフィルタ免除   | 信頼できる送信元 IP（許可リストとの違いに注意） |

#### 許可リスト と IP 許可リスト の違い（試験頻出）

```text
【Email Allowlist（メールの許可リスト）】
  → 特定 IP から送信されたメールをスパムフィルタをスキップして受信箱に届ける
  → メールアドレスや IP アドレスを指定

【IP Allowlist（IP 許可リスト）】
  → 受信ゲートウェイとして機能する IP アドレスを登録
  → この IP 経由で来たメールは Google が「既にスパムチェック済み」と認識
  → スパム分類を緩める効果がある（完全スキップではない）
```

#### フィッシング対策の詳細設定

```text
Admin コンソール → Gmail → スパム、フィッシング、マルウェア →
フィッシングと不正使用に対する保護（拡張前のスキャン）
```

| 設定                                             | 説明                                          |
| ------------------------------------------------ | --------------------------------------------- |
| **不正なリンクと画像に対して警告する**           | 不審な URL がある場合に警告バナーを表示       |
| **なりすましや認証されていないメールに注意**     | SPF/DKIM が未設定のドメインからのメールに警告 |
| **外部のナリスマシの可能性があるドメインを検出** | ドメイン名のタイポスクワッティングを検出      |
| **暗号化されていない添付ファイルから保護する**   | 暗号化スクリプトを含む添付ファイルを検出      |

#### ✅ ベストプラクティス：スパム・フィッシング対策

| #   | ベストプラクティス                                                                                                   |
| --- | -------------------------------------------------------------------------------------------------------------------- |
| 1   | **許可リストへの IP 登録は最小限に**。過度な許可はスパムフィルタを弱体化させる                                       |
| 2   | **受信ゲートウェイを設定した場合は、そのゲートウェイの IP のみを IP 許可リストに追加する**                           |
| 3   | **フィッシング対策の全設定を「迷惑メール隔離」ではなく「受信トレイへの警告」で開始し、誤検知を確認してから強化する** |
| 4   | **定期的にスパムレポートを確認し、許可/ブロックリストを見直す**                                                      |

### 1.6 Gmail の転送・POP/IMAP 設定

#### 🔰 転送・POP/IMAP の管理の重要性

ユーザーが Gmail のメールを外部のメールクライアントや別アカウントに転送・取得できる機能を管理します。セキュリティポリシーに応じて制限が必要です。

```text
Admin コンソール → Gmail → エンドユーザーアクセス
```

#### 各設定の説明と推奨値

| 設定名             | 説明                                               | セキュリティ推奨                         |
| ------------------ | -------------------------------------------------- | ---------------------------------------- |
| **POP アクセス**   | サードパーティのメールクライアントで Gmail を取得  | 組織のポリシーに応じて無効化を検討       |
| **IMAP アクセス**  | サードパーティのメールクライアントで Gmail を同期  | 組織のポリシーに応じて無効化を検討       |
| **自動転送**       | ユーザーが全メールを外部アドレスに自動転送する設定 | **無効化を強く推奨**（データ漏洩リスク） |
| **Gmail アドオン** | Gmail に追加機能をインストール                     | 承認済みアドオンのみ許可                 |

#### ✅ ベストプラクティス：転送・POP/IMAP

- **自動転送（外部へのメール転送）は原則禁止**。個人の Gmail などへの転送は情報漏洩の主要な経路
- **やむを得ず転送が必要なケースには、ルールベースの転送設定を使用**し、承認フローを設ける
- **POP/IMAP を無効化する場合はユーザーへの事前通知が必須**（Outlook など使用中のユーザーが影響を受ける）

> 🔗 **参考**: <https://support.google.com/a/answer/105694>

### 1.7 Google 推奨のメールセキュリティ（SPF・DKIM・DMARC）★最重要

#### これが試験の最頻出トピックです

メールの認証技術 SPF、DKIM、DMARC は、なりすましメール（フィッシング）の防止に不可欠です。これらが「なぜ必要か」「どのように機能するか」「どのように設定するか」を完全に理解してください。

---

#### SPF（Sender Policy Framework）

**SPF** は「このドメインからメールを送信できる IP アドレスはこれです」とDNSに宣言する仕組みです。

```text
【SPF の仕組み】

送信者: info@example.com（ Google のサーバーから送信）
                ↓
受信者のサーバー:「example.com の SPF レコードを確認」
                ↓
DNS TXT レコード: "v=spf1 include:_spf.google.com ~all"
                ↓
「_spf.google.com に含まれる IP から送られている → OK（PASS）」

もし Google 以外のサーバーから送られた場合:
「SPF レコードに含まれない IP → FAIL または SOFTFAIL」
→ スパムとして処理される可能性がある
```

**SPF レコードの書き方**:

```dns
# 基本形
v=spf1 include:_spf.google.com ~all

# Google + サードパーティ（例：Salesforce）も使う場合
v=spf1 include:_spf.google.com include:_spf.salesforce.com ~all

# 終端のポリシー（修飾子）の意味
-all  → 一致しないメールを「FAIL」（厳格：拒否）
~all  → 一致しないメールを「SOFTFAIL」（緩和：スパム扱い）
?all  → 一致しないメールを「NEUTRAL」（何もしない）
```

**SPF の設定手順（Google Workspace）**:

```text

1. DNS レジストラにログイン
2. ドメインの DNS 設定を開く
3. 新しい TXT レコードを追加:
   - ホスト名: @ （または空欄）
   - 値: v=spf1 include:_spf.google.com ~all
   - TTL: 3600（または DNS の設定に従う）
4. 変更を保存し、DNS の伝播を待つ
5. Admin Toolbox で確認: https://toolbox.googleapps.com/apps/checkmx/

```

> ⚠️ **SPF の制限**: SPF レコードには **最大 10 回のDNS 参照** という制限があります。サードパーティのメール送信サービスを多数追加すると、この制限を超えてしまうことがあります。

---

#### DKIM（DomainKeys Identified Mail）

**DKIM** は、送信メールに**電子署名**を付与し、受信者が「本当にそのドメインから送られたメールであり、改ざんされていない」ことを確認できる仕組みです。

```text
【DKIM の仕組み】

送信者（Gmail）:
  ┌─────────────────────────────────────────┐
  │ メール本文・ヘッダーのハッシュ値を計算   │
  │ 秘密鍵（Google が管理）で署名を作成     │
  │ メールのヘッダーに署名を追加して送信    │
  └─────────────────────────────────────────┘
              ↓ メールが届く
受信者のサーバー:
  ┌─────────────────────────────────────────┐
  │ DNS から example.com の公開鍵を取得    │
  │ 公開鍵でメールの署名を検証             │
  │ 検証成功 → 改ざんなし、正規ドメインからの送信 │
  └─────────────────────────────────────────┘
```

**DKIM の設定手順（Google Workspace）**:

```text
Step 1:
Admin コンソール → アプリ → Google Workspace → Gmail →
メールの認証 → DKIM キーを生成

Step 2:
生成された TXT レコードをコピー
（例: google._domainkey.example.com に追加する値）

Step 3:
DNS レジストラで TXT レコードを追加:
  ホスト名: google._domainkey.example.com
  値: v=DKIM1; k=rsa; p=MIGfMA0GCSQ... (長い公開鍵)
  TTL: 3600

Step 4:
Admin コンソールに戻り「DKIM を開始」をクリック
→ Google が秘密鍵で署名を開始

Step 5:
数時間後、Admin Toolbox で DKIM が正しく機能しているか確認
```

> ⚠️ **重要**: DKIM キーは **2048 ビット** を使用することを推奨します（1024ビットより安全）。2048ビットキーのDNSへの追加方法はDNSレジストラによって異なります（CNAMEレコードの使用が必要な場合あり）。

---

#### DMARC（Domain-based Message Authentication, Reporting, and Conformance）

**DMARC** は、SPF と DKIM の検証結果に基づいて、認証に失敗したメールをどう処理するかを受信者に伝え、かつレポートを受け取る仕組みです。

```text
【DMARC の動作フロー】

受信メール
    ↓
SPF チェック: PASS / FAIL
DKIM チェック: PASS / FAIL
    ↓ どちらか一方が PASS で「アライメント」が取れていれば認証成功
DMARC ポリシーを確認（DNS の _dmarc.example.com の TXT レコード）
    ↓ 認証失敗の場合
ポリシーに従って処理:
  p=none      → 何もしない（レポートのみ）
  p=quarantine → スパムフォルダへ移動
  p=reject    → メールを完全に拒否（バウンス）
    ↓
レポートを rua（集計レポート）や ruf（フォレンジックレポート）に送信
```

**DMARC レコードの書き方**:

```dns
# 最小構成（監視モード）
v=DMARC1; p=none; rua=mailto:dmarc-reports@example.com

# 隔離モード（認証失敗をスパムに）
v=DMARC1; p=quarantine; pct=100; rua=mailto:dmarc-reports@example.com

# 拒否モード（最も厳格）
v=DMARC1; p=reject; pct=100; rua=mailto:dmarc-reports@example.com; adkim=s; aspf=s

# 主要なタグの説明
p=       → ポリシー（none / quarantine / reject）
pct=     → ポリシーを適用するメールの割合（0-100%）
rua=     → 集計レポートの送信先メールアドレス
ruf=     → フォレンジック（詳細）レポートの送信先
adkim=   → DKIM アライメントの厳格度（r=緩和, s=厳格）
aspf=    → SPF アライメントの厳格度（r=緩和, s=厳格）
```

**DMARC の段階的導入手順（Google 推奨）**:

```text
Phase 1: 監視（最低1〜2週間）
  p=none; rua=mailto:dmarc-reports@example.com
  → レポートを受け取り、正規のメール送信元をすべて把握する

Phase 2: 段階的に隔離へ移行
  p=quarantine; pct=10  → 10% のみ隔離
  p=quarantine; pct=50  → 50% に拡大
  p=quarantine; pct=100 → 全メールに適用

Phase 3: 完全拒否（目標）
  p=reject; pct=100
  → 認証に失敗したメールを完全に拒否
```

---

#### SPF・DKIM・DMARC の全体的な関係図

```text
┌─────────────────────────────────────────────────────────┐
│  送信者（example.com）                                    │
│  ┌─────────────────────────────────────────────────┐    │
│  │ SPF: どの IP からの送信を許可するかを DNS に宣言  │    │
│  │ DKIM: メールに電子署名を付与                     │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
                         ↓ メール送信
┌─────────────────────────────────────────────────────────┐
│  受信者のサーバー                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │ 1. SPF 検証: 送信元 IP が DNS の宣言と一致するか │    │
│  │ 2. DKIM 検証: 電子署名が正しいか（改ざんなし）  │    │
│  │ 3. DMARC 確認: 検証結果に基づいてどう処理するか │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

三つが揃って初めて完全なメール認証が実現する
SPF のみ   → 送信元 IP の偽装は防げるが、メール本文の改ざんは防げない
DKIM のみ  → 改ざんは防げるが、別 IP からの送信は防げない
DMARC なし → 認証失敗時の処理ポリシーがなく、フィッシングが届く可能性がある
```

#### ✅ ベストプラクティス：SPF・DKIM・DMARC

| #   | ベストプラクティス                                                                   |
| --- | ------------------------------------------------------------------------------------ |
| 1   | **SPF・DKIM・DMARC を必ず3つセットで設定する**（1つや2つだけでは不完全）             |
| 2   | **DMARC は `p=none` から段階的に導入し、レポートを確認しながら `p=reject` を目指す** |
| 3   | **DMARC レポートは DMARC 分析ツール**（Postmark、dmarcian など）**で可視化する**     |
| 4   | **Salesforce、Mailchimp などサードパーティ送信サービスの IP も SPF に含める**        |
| 5   | **DKIM キーは 2048 ビットを使用し、年に一度ローテーションする**                      |

> 🔗 **参考（SPF）**: <https://support.google.com/a/answer/33786>
> 🔗 **参考（DKIM）**: <https://support.google.com/a/answer/174124>
> 🔗 **参考（DMARC）**: <https://support.google.com/a/answer/2466580>

### 1.9 Gmail アクセスの委任

#### 🔰 Gmail アクセスの委任とは

**Gmail の委任**は、ユーザーが自分の Gmail へのアクセスを別のユーザー（秘書、アシスタントなど）に許可する機能です。

```text
【委任の仕組み】

経営者（A さん）
  └── 秘書（B さん）に Gmail アクセスを委任

B さんが A さんの Gmail にアクセスできること:
  ✅ A さんのメールを読む
  ✅ A さんの名前でメールを送信する
  ✅ A さんのメールを削除する

B さんができないこと:
  ❌ A さんの Google アカウント設定の変更
  ❌ パスワードのリセット
  ❌ チャット・ドライブへのアクセス（メールのみ）
```

**管理者による委任の有効化**:

```text
Admin コンソール → Gmail → ユーザーの設定 → メールの委任
→ 「ユーザーがメールを他のユーザーに委任できるようにする」を有効化
```

**ユーザーによる委任の設定**:

```text
Gmail → 設定（歯車アイコン）→ すべての設定を表示 → アカウント →
メールの委任設定 → 別のアカウントを追加
```

> ⚠️ **重要**: 委任設定はユーザー自身が行います。管理者は委任機能の有効化/無効化のみを制御できます。
>
> 🔗 **参考**: <https://support.google.com/a/answer/1247360>

## Chapter 8: Section 2 試験対策まとめ

### 8.1 試験頻出パターン

#### パターン①：SPF・DKIM・DMARC の設問

```text
【典型的な問題文】
「組織のドメインからなりすましメールが送信されている報告がある。
 これを防ぐためにどの設定を行うべきか？」

【正解への考え方】
なりすましメール対策 = SPF + DKIM + DMARC の三点セット

✅ SPF: 送信許可 IP をDNSに宣言
✅ DKIM: 電子署名でメールの真正性を証明
✅ DMARC: 認証失敗時の処理ポリシーと報告

【DMARC の段階的導入が鍵】
p=none → p=quarantine → p=reject の順に実施
いきなり p=reject にすると正規メールまでブロックされる可能性がある
```

#### パターン②：Gmail ルーティング設定の問題

```text
【典型的な問題文】
「組織は現在 Exchange と Google Workspace を並行して使用している。
 特定の部門のメールは Exchange に、残りは Gmail に振り分けたい。」

【正解】
スプリット配信（Split Delivery）を設定する
  → 特定のユーザーは Exchange に転送
  → 残りのユーザーは Gmail で受信
```

#### パターン③：Drive の外部共有制御

```text
【典型的な問題文】
「財務部門のドライブファイルは一切外部と共有できないようにしたい。
 その他の部門は外部共有を許可したい。最適な設定方法は？」

【正解】

1. ルート組織全体で外部共有を「許可」（デフォルト）
2. 財務部門の OU を作成
3. 財務部門 OU の Drive 共有設定で外部共有を「禁止」（上書き）

または（Enterprise の場合）
Drive 信頼ルールで財務部門 OU からの外部共有をすべてブロック
```

#### パターン④：Google Meet のセキュリティ問題

```text
【典型的な問題文】
「Google Meet の会議に不審な外部参加者が入ってくる事件が発生した。
 これを防ぐための設定はどれか？」

【正解】
Meet の安全設定 → ノックイン機能を有効化
  → 組織外のユーザーがホストの承認なしに参加できなくなる
```

#### パターン⑤：Chat コンプライアンスの問題

```text
【典型的な問題文】
「法的調査のため、過去6ヶ月のチャットメッセージを提出する必要がある。
 しかしチャット履歴がオフになっていた。どうすべきだったか？」

【正解（予防策）】
チャット履歴を「オン」に設定し、Vault で保持ルールを設定する
  → 履歴がオンでないと、Vault での eDiscovery は不可能
  → コンプライアンス要件がある組織は必ずオンに
```
