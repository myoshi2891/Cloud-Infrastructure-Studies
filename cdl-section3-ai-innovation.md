## 2. AI・ML の基礎概念を理解する

### 2.1 AI・ML・DL・生成 AI の包含関係（最重要）

試験で最も基本的かつ頻出の概念です。
**4 つの技術が「入れ子構造（包含関係）」**になっていることを理解してください。

```
┌─────────────────────────────────────────────────────┐
│  AI（人工知能 / Artificial Intelligence）            │
│  「人間の知的行動をコンピュータで模倣する」           │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │  ML（機械学習 / Machine Learning）             │  │
│  │  「データからパターンを自動学習する」            │  │
│  │                                               │  │
│  │  ┌───────────────────────────────────────┐   │  │
│  │  │  深層学習（Deep Learning）              │   │  │
│  │  │  「多層ニューラルネットワークを使う」    │   │  │
│  │  │                                       │   │  │
│  │  │  ┌───────────────────────────────┐   │   │  │
│  │  │  │  生成 AI（Generative AI）       │   │   │  │
│  │  │  │  「新しいコンテンツを生成する」  │   │   │  │
│  │  │  │                               │   │   │  │
│  │  │  │  ┌───────────────────────┐   │   │   │  │
│  │  │  │  │ LLM（大規模言語モデル） │   │   │   │  │
│  │  │  │  │ テキスト生成に特化     │   │   │   │  │
│  │  │  │  └───────────────────────┘   │   │   │  │
│  │  │  └───────────────────────────────┘   │   │  │
│  │  └───────────────────────────────────────┘   │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

#### 各概念の定義と具体例

| 概念 | 定義 | 具体例 |
| --- | --- | --- |
| **AI（人工知能）** | 人間の知的行動をコンピュータで模倣する技術全般 | チェスプログラム・Siri・自動運転 |
| **ML（機械学習）** | データからパターンを自動学習する AI の手法 | スパムフィルタ・需要予測・顧客セグメント |
| **深層学習（DL）** | 多層のニューラルネットワークを使った ML | 画像認識・音声認識・翻訳 |
| **生成 AI** | 新しいテキスト・画像・音声・動画を生成できる AI | Gemini・Imagen・Veo |
| **LLM** | テキスト生成に特化した超大規模な言語モデル | Gemini・GPT-4・Claude |

> ⚠️ **試験の落とし穴**: 「生成 AI = LLM」は**誤り**。LLM はテキスト生成専門だが、
> 生成 AI には画像生成（Imagen）・動画生成（Veo）・音楽生成なども含まれる。

### 2.2 なぜ AI は今これほど注目されているのか

```
AI ブレークスルーの 3 つの要因:

要因 1: データ量の爆発的増加
  → スマートフォン・IoT・Web サービスの普及で
    学習に使えるデータが指数関数的に増加

要因 2: 計算能力の劇的向上
  → GPU・TPU の登場で大規模モデルの学習が現実的に
  → Google の TPU は AI 学習専用設計

要因 3: アルゴリズムの革新
  → 2017年: Google の Transformer アーキテクチャ発表
  → 現在の LLM（ChatGPT・Gemini等）はすべて Transformer ベース

Google の AI における「世界初」の実績:
  ✅ 2012年: ニューラルネットが猫を認識（Google Brain）
  ✅ 2015年: TensorFlow をオープンソース化
  ✅ 2016年: AlphaGo が囲碁世界チャンピオンを破る
  ✅ 2017年: Transformer アーキテクチャを発表（現代 LLM の基盤）
  ✅ 2020年: AlphaFold でタンパク質構造予測に革命
  ✅ 2023年: Gemini・Bard の登場
```

### 2.3 AI プロジェクト成功の鍵

```
AI の品質は「データの品質」に直結する（GIGO の法則）

GIGO = Garbage In, Garbage Out
「ゴミを入れればゴミが出る」

高品質な AI を作るための条件:
  ✅ 十分な量のデータ（学習データが少ないと精度が出ない）
  ✅ 高品質なデータ（ノイズ・誤りが少ない）
  ✅ 多様なデータ（バイアスが少ない）
  ✅ ラベル付きデータ（教師あり学習の場合）
  ✅ 目的に関連したデータ（関係のないデータは不要）
```

## 3. 機械学習の 3 つのアプローチ

### 3.1 教師あり学習（Supervised Learning）

```
定義: 正解ラベルが付いたデータを使って学習する手法

イメージ: 「先生」（正解ラベル）が「生徒」（モデル）を指導する

データの形式:
  入力（X）          正解ラベル（Y）
  ────────────────────────────────
  メールのテキスト →  スパム / 非スパム
  画像            →  猫 / 犬 / 鳥
  過去の売上データ →  来月の売上額
  患者データ      →  疾患あり / なし

タスクの種類:
  ┌─────────────────────────────────────────┐
  │ 分類（Classification）                  │
  │  → 離散的なカテゴリを予測               │
  │  例: メールがスパムか否か・画像の物体種別 │
  │                                         │
  │ 回帰（Regression）                      │
  │  → 連続的な数値を予測                   │
  │  例: 明日の気温・来月の売上・不動産価格  │
  └─────────────────────────────────────────┘
```

#### 教師あり学習のビジネス活用例

| 業界 | ユースケース | 入力データ | 予測対象 |
| --- | --- | --- | --- |
| 金融 | 与信審査 | 年収・勤続年数・取引履歴 | 返済可能か |
| EC | 購買予測 | 閲覧・購買履歴 | 購入確率 |
| 医療 | 疾患診断支援 | 検査データ・画像 | 疾患の有無 |
| 製造 | 品質検査 | センサーデータ・画像 | 不良品か否か |
| マーケティング | 離脱予測 | 顧客行動ログ | 解約確率 |

### 3.2 教師なし学習（Unsupervised Learning）

```
定義: 正解ラベルなしのデータからパターン・構造を自動発見する手法

イメージ: 「先生」なしで「生徒」が自分でルールを見つける

データの形式:
  入力（X）          ← 正解ラベルは存在しない！
  ────────────────────────────────────────────
  顧客の購買データ → （AI が自動的にグループを発見）
  ウェブのクリックログ → （AI がパターンを自動抽出）

タスクの種類:
  ┌──────────────────────────────────────────────┐
  │ クラスタリング（Clustering）                  │
  │  → 似たものをグループにまとめる               │
  │  例: 顧客を購買パターンで自動分類              │
  │                                              │
  │ 次元削減（Dimensionality Reduction）         │
  │  → 複雑なデータをシンプルな形に圧縮           │
  │  例: 高次元の特徴量を 2D に圧縮して可視化     │
  │                                              │
  │ 異常検知（Anomaly Detection）                │
  │  → 通常と異なるパターンを発見                │
  │  例: ネットワーク侵入検知・不正アクセス検知   │
  └──────────────────────────────────────────────┘
```

#### 教師なし学習のビジネス活用例

| ユースケース | 説明 | 効果 |
| --- | --- | --- |
| **顧客セグメンテーション** | 購買パターンで自動的に顧客グループを発見 | ターゲットマーケティングの精度向上 |
| **不正検知** | 正常な取引パターンから逸脱したものを検知 | 事前定義不要で新種の不正も検知 |
| **レコメンドエンジン** | 「この商品を見た人はこれも見ている」を発見 | 顧客体験向上・クロスセル強化 |
| **文書分類** | 大量のドキュメントを内容でグループ化 | 情報管理・ナレッジ整理 |

### 3.3 強化学習（Reinforcement Learning）

```
定義: エージェントが環境と対話しながら「報酬の最大化」を学習する手法

イメージ: 「ゲームを攻略するAI」のように、試行錯誤を繰り返して学習する

仕組み:
  ┌────────────────────────────────────────────────┐
  │ エージェント（AI）                              │
  │    ↓ 行動（Action）                           │
  │ 環境（Environment）                            │
  │    ↓ 状態の変化 + 報酬（Reward）               │
  │ エージェントが「報酬が増える行動」を学習する     │
  └────────────────────────────────────────────────┘

具体例（ゲーム攻略 AI）:
  行動: 上下左右に移動
  報酬: スコアが上がる → +10点、ゲームオーバー → -100点
  学習: 報酬が最大化される行動パターンを自動発見

ビジネス活用例:
  ✅ RLHF（人間フィードバックによる強化学習）
     → LLM の出力品質を向上させる（ChatGPT・Gemini も活用）
  ✅ 自動運転の制御ロジック
  ✅ ロボットの動作制御
  ✅ 広告入札の自動最適化
  ✅ データセンターの電力管理最適化
```

### 3.4 3 つのアプローチの比較まとめ

| 比較項目 | 教師あり学習 | 教師なし学習 | 強化学習 |
| --- | --- | --- | --- |
| **データ形式** | 入力 + 正解ラベル | 入力のみ（ラベルなし） | 環境との対話 |
| **学習方法** | 正解を教えて学習 | 自分でパターン発見 | 報酬を最大化して学習 |
| **代表タスク** | 分類・回帰 | クラスタリング・異常検知 | 最適制御・ゲーム |
| **GCP 活用例** | AutoML・Vision API | BigQuery ML のクラスタリング | 自動運転・RLHF |
| **必要なデータ量** | 大量（ラベル付き） | 大量（ラベルなしでOK） | 環境とのインタラクション |

> 📎 **参照**:  
> https://cloud.google.com/vertex-ai/docs/tabular-data/classification-regression/overview  
> https://cloud.google.com/vertex-ai/docs/anomaly-detection/overview

## 4. Google Cloud AI サービスの階層構造

### 4.1 AI サービスの 3 層アーキテクチャ

Google Cloud の AI サービスは、**使いやすさとカスタマイズ性のバランス**で
3 つの層に分かれています。

```
            使いやすさ ◄──────────────────────── カスタマイズ性
            ─────────────────────────────────────
            最大        中程度          最大

   Layer 1  Layer 2     Layer 3       Layer 4
  ┌────────┐ ┌────────┐ ┌──────────┐ ┌─────────────────────┐
  │プリビルト│ │AutoML  │ │Vertex AI │ │  生成 AI サービス    │
  │ AI API │ │（ノー  │ │（フル    │ │  （Gemini・          │
  │        │ │コード）│ │コード）  │ │   Vertex AI Studio） │
  └────────┘ └────────┘ └──────────┘ └─────────────────────┘
     ↑           ↑           ↑                  ↑
  ML知識不要   ML知識少し   ML専門知識         ビジネス活用重視
  コード最小   必要        必要
  即座に利用   数時間で    数週間〜数ヶ月       即座〜数時間
  カスタム不可 カスタム可   フルカスタム
  低コスト     中コスト     高コスト

どのレベルから始めるか:
  → まずプリビルト AI API を試す
  → 精度が不十分なら AutoML
  → さらに高い要件なら Vertex AI
```

## 5. プリビルト AI API（事前学習済みモデル）

### 5.1 プリビルト AI API とは

**プリビルト AI API** は、Google が膨大なデータで事前に学習済みのモデルを
**API（Application Programming Interface）として公開したサービス**です。

```
プリビルト AI API のメリット:
  ✅ ML の専門知識不要（API を呼び出すだけ）
  ✅ 即座に利用開始できる
  ✅ 学習コスト・時間が不要
  ✅ Google の高品質モデルを手軽に活用
  ✅ スケーリング・インフラ管理が不要

デメリット:
  ❌ 自社固有のドメインへのカスタマイズは限定的
  ❌ 業界特有の専門用語・表現に弱い場合がある
  ❌ 競合との差別化は難しい
```

### 5.2 主要なプリビルト AI API 詳細解説

#### ① Cloud Vision API（画像認識）

```
できること:
  - ラベル検出: 画像に何が写っているか自動タグ付け
  - OCR（文字認識）: 画像・スキャン画像のテキストを抽出
  - 顔検出: 感情・年齢・顔の向きを分析
  - 物体検出: 複数の物体と位置を同時検出
  - ランドマーク認識: 観光地・建物を自動識別
  - 成人向けコンテンツ検出: 不適切コンテンツの自動フィルタ
  - ロゴ認識: ブランドロゴの自動識別

ビジネスユースケース:
  ✅ EC サイトの商品画像の自動タグ付け（検索性向上）
  ✅ 製造ラインの外観検査自動化
  ✅ 医療画像の初期スクリーニング支援
  ✅ 書類・伝票・レシートのデジタル化（OCR）
  ✅ SNS の不適切コンテンツ自動検出・削除

API 呼び出しイメージ:
  画像ファイル → Cloud Vision API → {"labels": ["cat", "pet", "animal"],
                                    "text": "...", "faces": [...]}
```

#### ② Cloud Natural Language API（自然言語処理）

```
できること:
  - 感情分析（Sentiment Analysis）: テキストのポジ・ネガ・中立を判定
  - エンティティ抽出（Entity Extraction）: 人名・地名・組織名を抽出
  - 構文解析（Syntax Analysis）: 品詞・文の構造を解析
  - コンテンツ分類（Content Classification）: 700 以上のカテゴリに自動分類
  - エンティティ感情分析: 特定の人物・製品に対する感情を分析

ビジネスユースケース:
  ✅ カスタマーレビューの自動感情分析・品質改善
  ✅ ソーシャルリスニング（SNS でのブランド評判監視）
  ✅ サポートチケットの自動カテゴリ分類・優先度付け
  ✅ ニュース・文書の自動タグ付け・整理
  ✅ 競合他社への言及を自動抽出

例（感情分析の結果）:
  入力: 「この商品は品質が素晴らしく、配送も速くて大満足です」
  出力: {"score": 0.95, "magnitude": 1.8, "sentiment": "POSITIVE"}
```

#### ③ Cloud Translation API（翻訳）

```
できること:
  - 130 以上の言語間で高品質な機械翻訳
  - 自動言語検出（何語か自動判定）
  - ドキュメント翻訳（PDF・Word のレイアウトを保ちながら翻訳）
  - Glossary（用語集）でブランド固有の翻訳ルールを適用

ビジネスユースケース:

## 6. AutoML（ノーコード ML）

### 6.1 AutoML とは

**AutoML** は、ML の専門知識がなくても、
**独自データからカスタム ML モデルを構築できる**サービスです。

```
AutoML の革命的な点:

従来のカスタムML開発:
  データ収集 → 前処理 → 特徴量エンジニアリング → 
  モデル選択 → ハイパーパラメータ調整 → 評価 → デプロイ
  （MLエンジニアが数週間〜数ヶ月かけて実施）

AutoML:
  データをアップロード → トレーニング開始ボタン → モデル完成
  （数時間〜1日、ML知識不要）
  
  Google の AutoML が内部で自動実行:
  ✅ 最適なモデルアーキテクチャの自動選択
  ✅ ハイパーパラメータの自動チューニング（Neural Architecture Search）
  ✅ データ拡張（Data Augmentation）の自動適用
  ✅ クロスバリデーションによる過学習防止
```

### 6.2 AutoML の種類と用途

| AutoML サービス | 入力データ | 対応タスク | ビジネス活用例 |
| --- | --- | --- | --- |
| **AutoML Vision** | 画像 | 分類・物体検出・セグメンテーション | 製品不良品検出・医療画像診断 |
| **AutoML Natural Language** | テキスト | 分類・感情分析・エンティティ抽出 | カスタマーサポート自動分類 |
| **AutoML Tables** | 表形式データ（CSV） | 分類・回帰・予測 | 顧客離脱予測・需要予測 |
| **AutoML Video** | 動画 | 分類・物体追跡・アクション認識 | 品質管理・防犯カメラ分析 |
| **AutoML Translation** | 対訳テキスト | カスタム翻訳モデル | 業界固有の翻訳精度向上 |

### 6.3 AutoML の使い方（AutoML Tables を例に）

```
Step 1: データの準備・アップロード
  - CSV ファイル または BigQuery テーブルを指定
  - 最低 1,000 件（推奨: 10,000 件以上）
  - 例: 顧客ID・購買回数・最終購入日・解約済みフラグ

Step 2: 目的変数（ターゲット）の設定
  - 予測したい列を選択
  - 例: 「解約済みフラグ」（true/false）

Step 3: トレーニング実行
  - 予算（最大トレーニング時間）を設定
  - 「トレーニング開始」ボタンをクリック
  - Google が内部で最適なモデルを自動探索

Step 4: モデルの評価
  - 精度・再現率・AUC など自動計算
  - 特徴量重要度（どの変数が予測に重要か）を確認

Step 5: デプロイ・予測 API
  - ボタン1つでエンドポイントとしてデプロイ
  - REST API で新しいデータの予測結果を取得

結果例:
  「顧客 A: 解約確率 87%（高リスク！）→ リテンション施策を実施」
  「顧客 B: 解約確率 12%（低リスク）→ 通常フォロー」
```

### 6.4 AutoML vs プリビルト AI API vs Vertex AI

```
選択のポイント:

プリビルト AI API を使う:
  ✅ 汎用的なタスク（画像ラベル付け・感情分析・翻訳）
  ✅ 自社独自のデータで学習させる必要がない
  ✅ 即座に使いたい
  例: 商品画像を Vision API で自動タグ付け

AutoML を使う:
  ✅ 自社固有のデータで精度を上げたい
  ✅ ML の専門知識がないが独自モデルが必要
  ✅ プリビルト API では精度が足りない
  例: 自社製品の不良品を検出するカスタムモデル

Vertex AI（カスタムモデル）を使う:
  ✅ 独自のモデルアーキテクチャが必要
  ✅ ML エンジニアが社内にいる
  ✅ 最高精度・最大柔軟性が必要
  ✅ 大規模な分散トレーニングが必要
  例: 研究機関がゼロから設計した創薬予測モデル
```

## 8. 生成 AI（Generative AI）の世界

### 8.1 生成 AI とは何か

**生成 AI**（Generative AI）とは、
**新しいコンテンツ（テキスト・画像・音声・動画・コード）を生成できる AI** のことです。

```
従来の AI（識別 AI） vs 生成 AI の違い:

従来の識別AI:
  入力 → 分類・予測（既存の答えを選ぶ）
  例: 「この画像は猫か犬か？」→「猫です」（分類）
  例: 「来月の売上は？」→「3,500 万円」（予測）

生成 AI:
  入力 → 新しいコンテンツを生成（存在しなかったものを作る）
  例: 「秋の紅葉の風景の詩を書いて」→ AI が詩を創作
  例: 「犬が宇宙服を着ている画像を生成して」→ AI が画像を生成
  例: 「このバグを修正するコードを書いて」→ AI がコードを生成
```

### 8.2 大規模言語モデル（LLM）の仕組み

```
LLM（Large Language Model）とは:
  - 数十億〜数兆のパラメータを持つ超大規模な言語モデル
  - 膨大なテキストデータで学習済み
  - テキストの生成・理解・翻訳・要約・質問応答ができる

LLM が学習していること（自己回帰的言語モデル）:
  「次のトークン（単語）を予測する」タスクで学習

  例:
  「東京は日本の___」→ 「首都」を予測
  「今日は天気が___」→ 「良い」を予測
  
  この予測タスクを数百億回繰り返すことで、
  言語のパターン・世界の知識・推論能力を獲得する

重要な概念:
  トークン: テキストを処理する単位（単語、部分単語）
  コンテキストウィンドウ: 一度に処理できる最大テキスト量
    例: Gemini 1.5 Pro は最大 100 万トークン
  パラメータ: モデルが学習した値の数（多いほど高性能だが重い）
  プロンプト: LLM への入力テキスト（指示文）
```

### 8.3 基盤モデル（Foundation Model）

```
基盤モデルとは:
  - 大量の多様なデータで事前学習した「汎用的な大規模モデル」
  - 特定タスクに特化していない → 多様なタスクに応用できる
  - ファインチューニングで特定ドメインに特化できる

なぜ「基盤」と呼ぶか:
  - 様々なアプリケーションの「土台（基盤）」として使えるから
  - 上に「アプリケーション層」を積み上げて使う

基盤モデル vs 従来のMLモデル:

  従来のMLモデル:         基盤モデル:
  特定タスク専用    →     汎用的・多目的
  1つのことだけ     →     翻訳・要約・コード・QA等
  小規模データ      →     数兆トークンのデータで学習
  タスクごとに作成  →     1つのモデルで多数のタスクに対応

マルチモーダル基盤モデル:
  - テキストだけでなく、画像・音声・動画も処理できる
  - 例: Gemini は「テキスト+画像+音声+動画」を統合処理
```

### 8.4 プロンプトエンジニアリング

```
プロンプトエンジニアリングとは:
  - LLM から望ましい出力を引き出すために「入力文（プロンプト）」を最適化する技術
  - モデルを再学習させずに性能を向上できる

効果的なプロンプトの要素:

  ① 明確な指示:
    悪い例: 「要約して」
    良い例: 「以下の記事を3つの箇条書きで要約してください。各項目は30字以内にすること」

  ② 文脈・背景情報:
    「あなたは10年以上のキャリアを持つマーケティング専門家です。
     以下の製品について10代向けのキャッチコピーを5つ提案してください。」

  ③ 出力形式の指定:
    「JSON 形式で答えてください: {"name": ..., "price": ..., "category": ...}」

  ④ Few-shot プロンプト（例示）:
    「以下の例に従ってください:
     入力: 「東京」→ 出力: 「日本の首都・人口1400万人」
     入力: 「大阪」→ 出力: ???」

プロンプトエンジニアリングの限界:
  ❌ 自社固有の専門知識は教えられない（ファインチューニングが必要）
  ❌ コンテキストウィンドウに収まる情報量に限界がある
  ❌ 最新情報は含まれない（学習データのカットオフ問題）
```

## 10. 生成 AI の活用技術（RAG・ファインチューニング・グラウンディング）

### 10.1 LLM の根本的な課題

```
LLM が抱える 4 つの根本的な課題:

① ハルシネーション（幻覚）:
   事実と異なる内容を自信を持って生成する
   例: 「田中一郎という小説家の代表作は〇〇です」（実在しない）

② 知識のカットオフ（Knowledge Cutoff）:
   学習データの締め切り日以降の情報を知らない
   例: 「最新の法改正の内容は？」→ 学習後の法改正を知らない

③ 非公開情報を知らない:
   自社の社内文書・独自データを知らない
   例: 「弊社の製品マニュアルについて説明して」→ 知らない

④ コンテキストの限界:
   長すぎるドキュメントは一度に処理できない

→ これらを解決する技術が RAG・グラウンディング・ファインチューニング
```

### 10.2 RAG（Retrieval-Augmented Generation）

```
RAG（検索拡張生成）とは:
  - LLM に「外部知識ベースから関連情報を検索して渡す」技術
  - LLM の知識を動的に拡張できる

RAG の仕組み:

  ユーザーの質問
      ↓
  ① 検索（Retrieval）:
     ベクターデータベースから関連文書を検索
  ↓
  ② コンテキスト拡張（Augmentation）:
     質問 + 検索した文書をまとめてプロンプトに追加
  ↓
  ③ 生成（Generation）:
     LLM が文書を参照しながら回答を生成
  ↓
  回答（根拠となる文書の引用付き）

RAG なし（通常の LLM）:
  質問: 「弊社の製品保証期間は？」
  回答: 「一般的な製品保証は1年です。」← 自社情報を知らない

RAG あり:
  ① 社内文書から「製品保証規定.pdf」を検索して取得
  ② LLM に「製品保証規定.pdf + 質問」を渡す
  ③ 回答: 「弊社の製品保証期間は購入日から3年間です。（保証規定 P.3 参照）」

RAG のメリット:
  ✅ ハルシネーションを大幅に低減（根拠ある回答）
  ✅ 最新情報・自社情報に対応（知識の動的更新）
  ✅ 回答に引用元を明示できる（信頼性・透明性）
  ✅ モデルの再学習不要（データ更新のコスト低）
  ✅ 機密データをモデルに学習させなくて良い

Google Cloud での RAG 実装:
  Vertex AI Search（最も簡単・プリビルト）
  Vertex AI RAG Engine（カスタム RAG パイプライン構築）
  Grounding with Google Search（リアルタイムWeb検索で最新情報補完）
```

### 10.3 グラウンディング（Grounding）

```
グラウンディングとは:
  - LLM の回答を「信頼できるデータソース」に根拠付ける技術
  - RAG の上位概念（RAG はグラウンディングの一実装方法）

グラウンディングの種類:

① ファーストパーティデータ（自社データ）でのグラウンディング:
   - 社内文書・製品マニュアル・FAQ・データベース
   - Vertex AI Search（エンタープライズ検索）を使用
   - セキュリティ重視（自社 VPC 内で処理）

② Google 検索でのグラウンディング（Grounding with Google Search）:
   - リアルタイムのウェブ情報を取り込む
   - 「今日の株価は？」「最新のニュースは？」に対応
   - 知識カットオフ問題を解決

③ サードパーティデータ:
   - 外部のデータベース・APIから情報を取得
   - 天気・株価・為替・ニュース等

ビジネス活用例:
  カスタマーサポートボット:
    → 自社製品マニュアル + FAQ でグラウンディング
    → 「製品の設定方法は？」に正確に回答できる

  投資リサーチアシスタント:
    → 最新ニュース + 財務データでグラウンディング
    → 「今日の市場動向は？」に最新情報で回答
```

### 10.4 ファインチューニング（Fine-tuning）

```
ファインチューニングとは:
  - 事前学習済み基盤モデルを「自社固有データ」で追加学習させる技術
  - モデルの「重み（パラメータ）」自体を更新する

なぜファインチューニングが必要か:
  - 特定ドメインの専門用語・文体を習得させたい
  - 特定フォーマットの出力を安定して生成させたい
  - 業界固有の知識を深く理解させたい

ファインチューニングの手法:

  ① Supervised Fine-tuning（SFT）:
     「入力 → 望ましい出力」の例示データセットで追加学習
     例: [{"input": "商品Aの不具合は？", "output": "商品Aの不具合は..."}]

  ② RLHF（人間フィードバックによる強化学習）:
     人間が「良い回答・悪い回答」を評価 → モデルが好ましい回答を学習
     ChatGPT・Gemini の品質向上に使用された主要技術

  ③ PEFT（パラメータ効率の良いファインチューニング）:
     LoRA・QLoRA など、少ないパラメータで効率的にファインチューニング
     GPU 資源を大幅に削減

ファインチューニングのコストと注意点:
  ❌ 高品質なデータセット作成に時間とコストがかかる
  ❌ 学習インフラ（GPU）のコストがかかる
  ❌ モデルの更新のたびに再ファインチューニングが必要
  ❌ 過学習（Overfitting）のリスク
```

### 10.5 3 技術の使い分け比較

| 比較項目 | RAG / グラウンディング | ファインチューニング | プロンプトエンジニアリング |
| --- | --- | --- | --- |
| **目的** | 最新・自社情報を反映 | ドメイン知識・文体の習得 | 出力の最適化 |
| **データ更新** | リアルタイム更新可能 | 再学習が必要 | 不要 |
| **コスト** | 中程度 | 高い（GPU 学習） | 低い |
| **専門知識** | エンジニアが必要 | ML エンジニアが必要 | 誰でも可 |
| **ハルシネーション対策** | ◎ 効果的 | △ 一部改善 | △ 限定的 |
| **最適場面** | 自社ナレッジQA・最新情報 | 特定文体・専門用語習得 | まず試すべき最初の手段 |
| **GCP サービス** | Vertex AI Search・RAG Engine | Vertex AI Training | Vertex AI Studio |

## 11. AI エージェントと自律型 AI

### 11.1 AI エージェントとは

**AI エージェント**とは、LLM が「ツールを使いながら自律的に複数ステップのタスクを実行できる」システムです。

```
LLM と AI エージェントの違い:

通常の LLM（受動的）:
  ユーザー: 「東京から大阪の新幹線を調べて」
  LLM:     「のぞみは約 2h30m で...（学習データに基づく一般的な回答）」
            ← Web 検索せず、リアルタイム情報なし

AI エージェント（能動的・自律的）:
  ユーザー: 「来週の水曜日、東京から大阪の新幹線を予約して」
  エージェント:
    ① カレンダーツールで来週水曜を確認 → 1月15日と判断
    ② 新幹線予約 API を呼び出してのぞみ 7 号を検索
    ③ 空席確認 → 希望の座席を発見
    ④ 予約 API を呼び出して予約実行
    ⑤ メールで予約確認を送信
    → 「のぞみ 7 号 08:33 発を予約しました。確認メールを送りました」

エージェントが持つ能力:
  - ツール（API・検索・DB・計算）を自律的に選択・実行
  - 複数ステップを自律的に計画・実行
  - 中間結果を見て次のステップを動的に決定
  - 人間に確認・エスカレーションするタイミングを判断
```

### 11.2 Vertex AI Agent Builder

```
Vertex AI Agent Builder とは:
  - AI エージェントを構築・デプロイ・管理するプラットフォーム
  - ノーコード（Agent Designer）からフルコード（ADK）まで対応
  - Google 製・サードパーティ・自社カスタムのツールと連携可能

構成コンポーネント:

  ① Agent Designer（ローコード UI）:
     - ビジュアルインターフェースでエージェントを設計
     - ノーコードでエージェントのフローを定義
  
  ② Agent Development Kit（ADK）:
     - Python・Java でカスタムエージェントをコーディング
     - オープンソース（GitHub で公開）
     - 複雑なロジック・カスタムツールの実装
  
  ③ Agent Engine（マネージドランタイム）:
     - デプロイ・スケーリング・監視を自動管理
     - セッション管理・長期メモリ（Memory Bank）
     - 実行ログ・トレースの可視化

エージェントが使えるツール:
  - Google Search（Web 検索）
  - Vertex AI Search（社内文書検索）
  - Code Interpreter（コード実行・計算）
  - 外部 API（REST/GraphQL）
  - データベース（BigQuery・Cloud SQL）
  - Google Workspace（Gmail・Calendar・Drive）
  - カスタム関数（自社定義の任意の処理）

マルチエージェント:
  - 複数の専門エージェントが協調して複雑なタスクを処理
  - 例: 「リサーチエージェント」+「執筆エージェント」+「校正エージェント」が
        連携してレポートを自動生成
```

### 11.3 Gemini for Google Workspace（生産性 AI）

```
Gemini for Workspace とは:
  - Gmail・Docs・Sheets・Slides・Meet に Gemini の AI 機能を直接組み込む
  - ビジネスユーザーが普段使うツールの中で AI を活用できる

各アプリでの機能:

  Gmail（メール）:
    ✅ 長いメールスレッドを自動要約
    ✅ 返信メールのドラフト自動生成
    ✅ トーン調整（丁寧・カジュアル・簡潔に）

  Google Docs（文書）:
    ✅ アウトラインからドキュメントを自動生成
    ✅ 既存文書の要約・翻訳・改善提案
    ✅ 社内ドキュメントへの質問応答

## 12. 責任ある AI（Responsible AI）

### 12.1 なぜ責任ある AI が必要か

```
AI が引き起こす可能性のある問題:

① 差別・偏見（Bias）:
   - AI が学習したデータに含まれる偏りがモデルに反映される
   - 例: 採用 AI が過去の採用データ（男性偏重）から学習 
         → 女性応募者を不当に低評価するバイアスが生まれる

② ハルシネーション（誤った情報の生成）:
   - AI が事実と異なる情報を自信を持って生成する
   - 例: 医療 AI が誤った治療法を提案する → 患者への危害

③ プライバシー侵害:
   - 学習データに含まれる個人情報が漏洩する
   - AI が個人情報を不適切に活用する

④ 悪用・武器化:
   - フェイクニュース・ディープフェイク・詐欺コンテンツの大量生成
   - サイバー攻撃の自動化

⑤ 透明性・説明可能性の欠如:
   - AI がなぜその判断をしたかを説明できない（ブラックボックス問題）
   - ローンの否認・人事評価に AI を使う場合、理由を説明できない

→ これらのリスクに対処するために「責任ある AI」の原則が必要
```

### 12.2 Google の AI 原則（7 原則）

Google は 2018 年に AI 開発の 7 つの原則を公表し、
全製品・研究に適用しています。

```

① 社会的に有益であること（Be socially beneficial）
   AI の潜在的なリスクより利益が大きい場合にのみ開発・利用する

② 不公平なバイアスの創出・強化を行わないこと
   人種・性別・宗教等による差別的な偏りを避ける

③ 安全性を考慮して構築・テストすること
   予期しない動作や悪用を防ぐ安全対策を組み込む

④ 人々に対して説明責任を果たすこと
   AI の判断を人間が理解・評価・覆せる仕組みを設ける

⑤ プライバシーの設計原則を組み込むこと
   データ収集の透明性・ユーザーのコントロール権を保証

⑥ 科学的卓越性の高い基準を維持すること
   AI の進歩のための多角的・開かれた議論を行う

⑦ 上記原則に沿って利用されるよう提供すること
   武器・監視・人権侵害への利用を禁止
```

### 12.4 ハルシネーション対策

```
ハルシネーション（AI の幻覚）とは:
  LLM が事実と異なる内容を、確信を持った形で生成する現象

ハルシネーションが起きる原因:
  - LLM は「次のトークンを確率的に予測」する仕組み
  - 「正確な事実」ではなく「もっともらしい文章」を生成する
  - 学習データに誤りがあるとそれを学習する

ハルシネーションの例:
  ❌ 「田中一郎氏（実在しない人物）は 2023 年にノーベル物理学賞を受賞しました」
  ❌ 「この薬の副作用は...（間違った情報）」
  ❌ 「このコードで動きます（実際は動かないコード）」

対策方法:
  ✅ RAG（外部知識ベースからの検索）で根拠ある回答を生成
  ✅ グラウンディング（Google Search との接続）で最新情報を参照
  ✅ Human-in-the-Loop（重要な決定に人間のレビューを挟む）
  ✅ 低 Temperature 設定（より保守的・一貫した回答）
  ✅ 回答の確信度スコアを表示して不確かな場合は明示
  ✅ ファクトチェック機能の組み込み
```

### 12.5 AI の公平性とバイアス

```
AI のバイアスの種類:

① 選択バイアス（Sampling Bias）:
   学習データが特定の集団に偏っている
   例: 医療 AI の学習データが白人男性中心 → 他の人種に精度が低い

② 確証バイアス（Confirmation Bias）:
   既存の偏見を強化する方向に学習が進む
   例: 採用 AI が「男性エンジニアが多い」現状を「正解」として学習

③ 測定バイアス（Measurement Bias）:
   データの収集・ラベリング方法自体に偏りがある
   例: アノテーターの文化的背景が正解ラベルに影響する

バイアス対策:
  ✅ 多様なデータセットの使用（人種・性別・文化の多様性を確保）
  ✅ Vertex AI の Model Evaluation で公平性指標を定期的に測定
  ✅ Sensitive Data Protection で差別的な属性の不適切な利用を検出
  ✅ 多様なバックグラウンドを持つチームでの AI 開発
  ✅ 外部の第三者による公平性監査
```

### 12.6 説明可能な AI（Explainable AI / XAI）

```
Explainable AI（XAI）とは:
  AI がなぜその判断をしたかを人間が理解できる形で説明する技術

なぜ説明可能性が必要か:
  - 規制対応: EU AI Act・金融規制で「判断根拠の説明」が義務化
  - 信頼性: ユーザーが AI を信頼して使えるようになる
  - デバッグ: 誤判断の原因を特定して改善できる
  - 公平性確認: バイアスの原因を特定できる

Vertex Explainable AI の機能:
  - 特徴量重要度（Feature Importance）:
    「この融資の否認は『収入』『負債比率』の影響が大きい」
  
  - 注意機構の可視化（Attention Visualization）:
    「この画像のどの部分を見て猫と判断したか」をヒートマップで表示
  
  - SHAP（Shapley Additive Explanations）:
    各特徴量が予測に与えた影響量を数値で表示

実際の活用例:
  銀行の融資審査 AI:
    「田中様のローン申請が否認された理由:
     ・収入が申請額に対して低い（重要度 60%）
     ・負債比率が高い（重要度 30%）
     ・勤続年数が短い（重要度 10%）」
  → 顧客への説明責任を果たせる
```

#### ✅ ベストプラクティス: 責任ある AI の実装

```yaml
設計フェーズ:
  - AI 倫理チェックリストを作成してすべてのプロジェクトで確認
  - 多様なバックグラウンドを持つチームで設計（ダイバーシティ）
  - ユースケースごとにリスクアセスメントを実施

開発フェーズ:
  - 学習データのバイアス分析を必ず実施
  - Vertex AI の公平性指標で複数の人口属性グループで評価
  - Model Card（モデルの仕様・限界・バイアス評価結果を記述した文書）を作成

デプロイ・運用フェーズ:
  - Human-in-the-Loop: 高リスク決定（医療診断・融資審査）には人間のレビューを必須化
  - 定期的なバイアス監査・公平性レビュー
  - AI 利用をユーザーに明示（「このメッセージは AI が生成しました」）
  - フィードバックループ: ユーザーの指摘を受けて継続的に改善
```

## 14. Section 3 総まとめ・頻出問題パターン

### 14.1 最重要用語の一問一答

```
Q: AI・ML・深層学習・生成 AI の包含関係は？
A: AI ⊃ ML ⊃ 深層学習 ⊃ 生成 AI ⊃ LLM の入れ子構造
   （AI が最も広い概念、LLM が最も狭い概念）

Q: 教師あり学習・教師なし学習・強化学習の違いは？
A: 教師あり=ラベル付きデータで正解を教えて学習
   教師なし=ラベルなしデータからパターンを自動発見
   強化学習=報酬を最大化する行動を試行錯誤で学習

Q: Gemini と Gemma の最大の違いは？
A: Gemini は Google が提供するクローズドモデル（API 利用）
   Gemma はオープンウェイト（自社サーバーで実行可能）

Q: RAG とファインチューニングの使い分けは？
A: RAG=最新情報・自社文書を参照（データ更新が容易）
   ファインチューニング=ドメイン知識・文体を習得させる（モデル再学習が必要）

Q: ハルシネーションとは何か？どう対処するか？
A: LLM が事実と異なる内容を自信を持って生成する現象
   対処: RAG・グラウンディング・Human-in-the-Loop

Q: プリビルト AI API と AutoML の使い分けは？
A: プリビルト API=自社固有データ学習不要な汎用タスク
   AutoML=自社固有データでカスタムモデルを構築（ノーコード）

Q: 基盤モデルとは何か？
A: 大量の多様なデータで事前学習した汎用的な大規模モデル
   ファインチューニングで特定ドメインに特化できる

Q: 責任ある AI の 6 原則は？
A: 公平性・信頼性と安全性・プライバシーとセキュリティ・
   包括性・透明性と説明可能性・説明責任

Q: SynthID とは何か？
A: Google DeepMind が開発した AI 生成コンテンツへの電子透かし技術
   人間の目には見えないが AI が検出できる

Q: RLHF とは何か？
A: Reinforcement Learning from Human Feedback（人間フィードバックによる強化学習）
   人間がAI の回答を評価し、その評価を基にモデルを改善する手法
```
