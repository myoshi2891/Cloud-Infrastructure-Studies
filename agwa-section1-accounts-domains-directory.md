## Chapter 1: ユーザーライフサイクルの管理（1.1）

### 1.1-A 移行戦略とツールの選定

#### 🔰 移行とは何か？

Google Workspace を初めて導入する際や、他のシステムから乗り換える際に、既存のメール・カレンダー・連絡先データを Google Workspace に移行する必要があります。移行元のシステムによって使用すべきツールが異なるため、まず**どのツールが何のための移行に使われるのか**を正確に把握することが重要です。

---

#### 主要な移行ツールの全比較表

| ツール名 | 移行元 | 移行対象データ | 規模感 | 特徴 |
| --------- | -------- | ------------- | -------- | ------ |
| **データ移行サービス**（Admin コンソール内） | Microsoft Exchange, Office 365, Gmail（他組織）, IMAP サーバー | メール、カレンダー、連絡先 | 小〜中規模 | GUIで操作、追加インストール不要 |
| **GWMME**（Google Workspace Migration for Microsoft Exchange） | Microsoft Exchange オンプレミス | メール、カレンダー、連絡先 | 中〜大規模 | オンプレミスに近い環境向けの詳細設定が可能 |
| **Google Workspace Migrate** | Exchange, SharePoint, Box, ファイルサーバー | メール、ファイル、カレンダー | 大規模・複雑 | オンプレミス型の高機能移行プラットフォーム |
| **Gmail のインポート機能** | IMAP サーバー（個人向け） | メールのみ | 個人 | エンドユーザーが自分で操作 |
| **GCDS**（Google Cloud Directory Sync） | Active Directory / LDAP | ユーザー・グループ情報のみ（メールデータは含まない） | 中〜大規模 | 継続的な同期（移行後も定期実行） |

---

#### ツール選定フローチャート

```
移行元はどのシステムですか？
│
├── Microsoft Exchange（オンプレミス）
│   ├── 規模が大きい or 複雑な要件
│   │   └── → Google Workspace Migrate
│   └── 標準的な移行
│       └── → GWMME または データ移行サービス
│
├── Microsoft 365 / Office 365
│   └── → データ移行サービス（Admin コンソール）
│
├── 別の Google Workspace ドメイン
│   └── → データ移行サービス（Admin コンソール）
│
├── Gmail（一般）
│   └── → データ移行サービス（Admin コンソール）
│
├── IMAP 対応サーバー（Lotus Notes 等）
│   └── → データ移行サービス（IMAP 接続）
│
└── Active Directory / LDAP のユーザー情報のみ
    └── → GCDS（メールデータの移行ではなく、ユーザー同期）
```

---

#### ✅ ベストプラクティス：移行戦略

| # | ベストプラクティス | 理由 |
| --- | ------------------- | ------ |
| 1 | **本番移行前にパイロットユーザー（10〜50名）でテスト実施** | 問題を事前に発見し、本番リスクを最小化する |
| 2 | **移行前にユーザーデータのバックアップを取得** | 万が一のデータ消失に備える |
| 3 | **移行は段階的に実施（部門ごと、地域ごと）** | 問題発生時の影響範囲を限定できる |
| 4 | **移行後は元システムを即座に廃止せず、一定期間並行稼働** | データの整合性確認とユーザーの適応期間を確保 |
| 5 | **移行ログを必ず確認し、エラーアイテムを記録・対処** | 見落としによるデータ欠損を防ぐ |

> 🔗 **参考**: https://support.google.com/a/answer/6374360  
> 🔗 **参考（GWMME）**: https://support.google.com/a/answer/180898  
> 🔗 **参考（Workspace Migrate）**: https://support.google.com/workspacemigrate

### 1.1-B ユーザーアカウントの手動作成

#### Admin コンソールでの手動作成手順

**Admin コンソール** → ディレクトリ → ユーザー → **新しいユーザーを追加**

```
設定する主要な項目：
1. 姓・名（後から変更可能）
2. プライマリメールアドレス（ユーザー名 + ドメイン）
3. 所属する組織部門（OU）
4. パスワード（自動生成 or 手動設定）
5. 次回ログイン時にパスワード変更を強制するか
6. ライセンスの割り当て
```

#### Admin コンソールでの一括作成 (CSV インポート)

大量のユーザーを作成する場合は **CSV インポート** が効率的です。

```csv
# CSV フォーマット例
First Name [Required],Last Name [Required],Email Address [Required],Password [Required],Password Hash Function [UPLOAD ONLY],Org Unit Path [Required],
山田,太郎,taro.yamada@example.com,ChangeMe123!,,/営業部,
鈴木,花子,hanako.suzuki@example.com,ChangeMe123!,,/開発部,
```

**手順**:

1. Admin コンソール → ユーザー → **その他のオプション** → ユーザーを一括アップロード
2. CSV テンプレートをダウンロードし、記入
3. アップロードして完了を確認

### 1.1-C 自動プロビジョニングとデプロビジョニング

#### 🔰 なぜ自動化が重要か

手動でのユーザー管理は、以下の問題を引き起こします：

- **入社処理の遅延**: IT担当者への依存で、初日から使えない状態になる
- **退職後もアカウントが残る**: セキュリティインシデントの原因
- **属性情報の不整合**: 異動・昇格時の更新漏れ

これらを解決するのが**自動プロビジョニング**です。

---

#### 方法①：GCDS（Google Cloud Directory Sync）

**GCDS** は、オンプレミスの **Active Directory (AD)** または **LDAP ディレクトリ** と Google Workspace を継続的に同期するツールです。

```
【GCDS の同期フロー】

Active Directory / LDAP（オンプレミス）
    ↓ GCDS が差分を検出
    ↓ シミュレーション実行（変更前に確認）
    ↓ 本番同期
Google Workspace（ユーザー・グループ・エイリアス）

同期方向：AD/LDAP → Google のみ（一方向）
         Google で変更した内容は AD に反映されない
```

**GCDS でできること**：

- ユーザーの作成・更新・削除（または停止）
- グループ・配信リストの同期
- カスタム属性のマッピング
- スケジュール実行（例：毎時自動同期）

**GCDS の設定手順概要**：

1. GCDS ツールをオンプレミスサーバーにインストール
2. AD/LDAP の接続情報（サーバーアドレス・認証情報）を設定
3. 同期するユーザー・グループの LDAP クエリを定義
4. Google Workspace 管理者アカウントと接続
5. **シミュレーション実行**で変更内容を事前確認
6. 本番同期を実行し、スケジュールを設定

> ⚠️ **重要**: 本番同期の前に必ず**シミュレーション（ドライラン）**を実施し、意図しないユーザー削除がないことを確認してください。

---

#### 方法②：サードパーティ IdP による自動プロビジョニング（SCIM）

**Okta**, **Azure AD（Entra ID）**, **OneLogin** などのアイデンティティプロバイダーと Google Workspace を連携することで、**SCIM プロトコル**を使用した自動プロビジョニングが実現できます。

```
【SCIM 連携の仕組み】

HR システム（Workday など）
    ↓ 人事情報の更新（入社・異動・退職）
IdP（Okta / Azure AD）
    ↓ SCIM プロトコル
Google Workspace
    ↓ ユーザー自動作成・更新・無効化
```

**メリット**：

- HR システムと連動した完全自動化
- Just-in-Time プロビジョニング（初回ログイン時に自動作成）
- 退職者の自動無効化

> 🔗 **参考（GCDS）**: https://support.google.com/a/answer/106368  
> 🔗 **参考（SCIM）**: https://support.google.com/a/answer/7614472

### 1.1-D サードパーティ IdP を使ったプロビジョニングと認証（SAML SSO）

#### 🔰 SAML SSO とは

**SAML（Security Assertion Markup Language）** は、アイデンティティプロバイダー（IdP）とサービスプロバイダー（SP）の間で認証情報を安全にやり取りするための標準規格です。

Google Workspace では以下の役割を担います：

- **SP（サービスプロバイダー）**: Google Workspace
- **IdP（アイデンティティプロバイダー）**: Okta、Azure AD、ADFS など

```
【SAML SSO の認証フロー（SP-initiated）】

ユーザー → Google Workspace へアクセス
                ↓ 未認証のためリダイレクト
           IdP（Okta など）のログイン画面
                ↓ ユーザーが認証
           SAML アサーション（認証証明書）を生成
                ↓ ブラウザ経由でGoogleへ送信
           Google Workspace がアサーションを検証
                ↓ 認証成功
           Google Workspace のサービスにアクセス可能
```

#### SAML SSO の設定手順（Admin コンソール）

```
Admin コンソール → セキュリティ → 認証 → サードパーティ IdP での SSO
```

**設定項目**：

| 項目 | 説明 |
| ------ | ------ |
| **SSO プロファイルを使用する** | 有効にする |
| **ログイン ページの URL** | IdP が提供するログインエンドポイント URL |
| **ログアウト ページの URL** | ログアウト後にリダイレクトされる URL |
| **パスワード変更 URL** | パスワード変更時のリダイレクト先 |
| **証明書のアップロード** | IdP の署名証明書（X.509 形式）をアップロード |
| **ネットワーク マスク** | SSO を適用しない特定の IP アドレス範囲 |

---

#### SP-initiated SSO vs IdP-initiated SSO の違い

| 比較項目 | SP-initiated SSO | IdP-initiated SSO |
| --------- | ----------------- | ------------------ |
| **起点** | ユーザーが Google にアクセス | ユーザーが IdP ダッシュボードからアプリを選択 |
| **フロー** | Google → IdP → Google | IdP → Google |
| **一般的な使用場面** | ブックマークや URL からのアクセス | 社内ポータルからのシングルクリックアクセス |
| **セキュリティ** | CSRF トークンで保護可能 | 若干リスクが高い（CSRF 攻撃への考慮が必要） |

---

#### ✅ ベストプラクティス：SAML SSO

| # | ベストプラクティス |
| --- | ------------------- |
| 1 | **証明書の有効期限を監視し、期限前に更新する**（証明書切れでSSO不可になる） |
| 2 | **スーパー管理者アカウントは SSO 対象から除外する**（IdP 障害時の緊急アクセス確保） |
| 3 | **テストユーザーで動作確認を行ってから全体適用する** |
| 4 | **ネットワークマスクを活用し、社内 IP からは直接ログインを許可する** |

> 🔗 **参考**: https://support.google.com/a/answer/60224  
> 🔗 **参考（SAML設定詳細）**: https://support.google.com/a/answer/6087519

### 1.1-E GCDS と Directory Sync の使い分け

#### 試験頻出：第一者同期ツールのユースケース

Google が提供する同期ツールには複数あり、それぞれのユースケースを正確に理解することが試験で問われます。

| ツール | 用途 | 同期方向 | 同期タイプ |
| -------- | ------ | --------- | ----------- |
| **GCDS**（Google Cloud Directory Sync） | オンプレミス AD/LDAP → Google Workspace | 一方向（AD → Google） | 継続的・定期的 |
| **Directory Sync**（クラウド版 GCDS） | 外部ディレクトリ（Active Directory/LDAP） → Google Workspace / Cloud Identity | 片方向のみ（外部ディレクトリ → Google） | オンデマンド |

#### GCDS が適しているケース

- オンプレミスの Active Directory が存在する企業
- 人事マスターが AD に格納されている
- Workspace への移行後も AD と並行運用する
- 毎時や毎日など定期的な自動同期が必要

#### Directory Sync が適しているケース

> Directory Sync は GCDS（Google Cloud Directory Sync）のクラウドベース後継版です。GCDS がオンプレミスで動作するのに対し、Directory Sync は Google のインフラ上でホスト・管理されます。

- Google Workspace と Google Cloud の ID を統合管理
- Cloud Identity ユーザーと Workspace ユーザーの同期

> 🔗 **参考（GCDS 詳細）**: https://support.google.com/a/answer/106368  
> 🔗 **参考（ベストプラクティス）**: https://support.google.com/a/answer/7177830

### 1.1-G アカウントの削除・停止・復元・アーカイブ

#### アカウント状態の違いと使い分け

これは試験で特に頻繁に問われる重要なポイントです。それぞれの状態を正確に理解してください。

```
【アカウント状態の遷移図】

アクティブ（Active）
    │
    ├─── 停止（Suspended）
    │       - アクセス不可
    │       - データは保持
    │       - ライセンスは消費したまま
    │       - いつでもアクティブに戻せる
    │
    ├─── 削除（Deleted）
    │       - アクセス不可
    │       - データは20日間保持
    │       - 20日以内なら復元可能
    │       - 20日後は完全削除
    │
    └─── アーカイブ（Archived）
            - アクセス不可（ログイン不可）
            - データは保持
            - アーカイブライセンスに変更でコスト削減
            - 主に退職者データの長期保持に使用
```

#### 各操作の詳細比較表

| 操作 | ユーザーのアクセス | データの保持 | ライセンス | 復元可否 | 主な用途 |
| ------ | -------------- | ----------- | ---------- | -------- | --------- |
| **停止（Suspend）** | ❌ 不可 | ✅ 保持 | 消費継続 | ✅ いつでも可能 | 長期休暇、調査中、一時停止 |
| **削除（Delete）** | ❌ 不可 | ⚠️ 20日間のみ | 解放される | ✅ 20日以内のみ | 退職者（データ不要の場合） |
| **アーカイブ（Archive）** | ❌ 不可 | ✅ 無期限保持 | アーカイブライセンス | ✅ 再アクティブ化可能 | 退職者（データ保持が必要な場合）|

#### ✅ ベストプラクティス：退職者処理のゴールデンフロー

```
Step 1: アカウントを即時【停止】（退職当日）
         → データは保持したままアクセスをブロック
         ↓
Step 2: Drive データの所有権を上司または後任者に【移転】
         → 業務継続性を確保
         ↓
Step 3: メールの転送設定（必要に応じて）
         → 退職者宛のメールを担当引継ぎ者へ転送
         ↓
Step 4: ライセンスの評価
         → データ保持が必要 → 【アーカイブ】ライセンスに変更
         → データ不要       → 【削除】（20日の猶予あり）
         ↓
Step 5: 一定期間後（30〜90日）に最終削除または長期アーカイブ
```

> 🔗 **参考（ユーザーの停止・削除）**: https://support.google.com/a/answer/33312  
> 🔗 **参考（アーカイブ）**: https://support.google.com/a/answer/9032727  
> 🔗 **参考（アカウント復元）**: https://support.google.com/a/answer/1247360

### 1.1-H Drive データの所有権移転

#### 所有権移転が必要な場面

- 退職者が作成したファイルの引き継ぎ
- 部署間の異動によるファイル管理者の変更
- 旧アカウントから新アカウントへの統合

#### Admin コンソールでの実施手順

```
Admin コンソール → ユーザー → [対象ユーザー] → 
アプリ → Google Workspace → Drive と Docs → 
データを移転
```

**設定項目**:

- 移転先ユーザー（新しい所有者）のメールアドレスを入力
- 移転範囲（マイドライブのみ / 共有ドライブは対象外）

#### 重要な制約事項

```
⚠️ 注意事項：

1. 移転できるのは【マイドライブ】内のファイルのみ
   共有ドライブのファイルは別途メンバー管理で対応

2. 移転先ユーザーは同じ組織内のユーザーのみ

3. Google フォームは移転不可
   （Forms のデータは別途エクスポートが必要）

4. 移転後、元のユーザーはファイルへのアクセスを失う
   （必要に応じて閲覧権限を付与し直す）
```

> 🔗 **参考**: https://support.google.com/a/answer/1247360

### 1.1-J パスワード管理

#### 管理者によるパスワード操作

| 操作 | 方法 | 使用シーン |
| ------ | ------ | ----------- |
| **パスワードリセット** | Admin コンソール → ユーザー → パスワードをリセット | ユーザーがパスワードを忘れた場合 |

| **パスワードポリシーの設定** | Admin コンソール → セキュリティ → パスワード | 最低文字数・強度を組織全体に適用 |

> ⚠️ **試験ポイント**: NIST の最新ガイドライン（SP 800-63B）では、定期的なパスワード変更の強制は**推奨されない**場合があります。侵害の証拠がある場合のみ変更を要求するアプローチが現在の主流です。

## Chapter 2: 組織部門（OU）の設計と作成（1.2）

### 2.1 OU（組織部門）とは

#### 🔰 OU の基本概念

**OU（Organizational Unit / 組織部門）** は、Google Workspace のユーザーやデバイスを階層的にグループ化するための「フォルダ」です。OU に設定したポリシーは、配下の全ユーザー・デバイスに**自動的に継承**されます。

```
【OU の階層と継承の仕組み】

ルート組織（example.com）← 最上位ポリシーを設定
    │
    ├── 本社          ← 本社固有のポリシーを追加・上書き可能
    │   ├── 営業部    ← さらに細かいポリシーを設定可能
    │   └── 開発部
    │       └── 研究チーム
    │
    ├── 支社A
    │   └── 営業部
    │
    └── 特殊アカウント（ポリシー例外用）
        ├── サービスアカウント
        └── 会議室リソース

【継承のルール】
- 上位 OU のポリシーは下位 OU に継承される
- 下位 OU で上書き（継承を切る）ことも可能
- ユーザーは必ず1つの OU に属する
```

---

### 2.2 OU の設計原則

#### なぜ OU 設計が重要か

OU の設計はポリシー管理の基盤です。設計が不適切だと、以下の問題が発生します：

- 一部のユーザーにのみ適用したいポリシーが全体に広がってしまう
- 例外対応のために複雑な設定が必要になる
- 管理が煩雑になり、設定ミスのリスクが増大する

#### Google 推奨の OU 設計パターン

**パターン1: 機能（役職・部門）ベース**

```
example.com
├── 一般社員
│   ├── 営業部
│   ├── 開発部
│   └── 管理部
├── 管理職
├── パートタイマー
└── 特権アカウント（管理者）
```

**パターン2: ポリシーの差異ベース（推奨）**

```
example.com（デフォルトポリシー）
├── 制限なし（外部共有OK、フルアクセス）
│   └── 経営層・上位管理職
├── 標準設定
│   ├── 一般社員
│   └── 派遣社員
├── 厳格設定（外部共有禁止）
│   └── 財務・法務・人事
└── 例外アカウント
    └── サービスアカウント・テストアカウント
```

#### OU 設計の重要な原則

| 原則 | 説明 |
| ------ | ------ |
| **ポリシー差異で分ける** | 同じポリシーが適用されるユーザーは同じ OU にまとめる |
| **深さは5階層以内** | 深すぎると継承関係が複雑になり管理困難になる |
| **例外 OU を用意する** | 特殊なポリシーが必要なユーザー用に例外 OU を設ける |
| **組織変更に強い設計** | 組織改編で OU を大幅に作り直さなくていい構造にする |
| **ユーザーと管理対象デバイスを分ける** | デバイスポリシーと ユーザーポリシーは別 OU で管理 |

---

### 2.3 OU の作成と管理の実際

#### Admin コンソールでの OU 作成手順

```
Admin コンソール → ディレクトリ → 組織部門 → ➕（組織部門を追加）
```

**入力項目**:

- 組織部門名
- 説明（任意）
- 親の組織部門（どの OU の配下に作成するか）

#### OU へのユーザーの移動

```
方法1: 個別移動
  Admin コンソール → ユーザー → [ユーザー] → 組織部門 → 編集

方法2: 一括移動（CSV）
  Admin コンソール → ユーザー → CSVでユーザーを更新
  CSV の「Org Unit Path」列に新しい OU パスを記入

方法3: API 経由（スクリプト）
  Directory API の patch メソッドで orgUnitPath を更新
```

#### OU でのポリシー管理

```
OU ごとに設定できるポリシー例：
✓ Gmail の機能（転送・POP/IMAP のオン/オフ）
✓ Drive の外部共有設定
✓ Meet の機能（録画の許可など）
✓ パスワード強度の設定
✓ セキュリティキーの要求
✓ アプリへのアクセス制限
```

#### ✅ ベストプラクティス：OU 管理

| # | ベストプラクティス |
| --- | ------------------- |
| 1 | **OU はシンプルに保つ**。設計しすぎると管理が複雑になる |
| 2 | **サービスアカウントや会議室は専用 OU に配置**してポリシーの例外管理を容易にする |
| 3 | **OU 移動はユーザーに事前通知する**（ポリシー変更で使えなくなる機能が生じる可能性がある） |
| 4 | **テストユーザー用の OU を設ける**。新しいポリシーは本番適用前にここでテストする |
| 5 | **OU の命名規則を標準化する**（例：`/Japan/Tokyo/Engineering`）|

> 🔗 **参考**: https://support.google.com/a/answer/182537  
> 🔗 **参考（OU設計のベストプラクティス）**: https://support.google.com/a/answer/4352075

## Chapter 3: グループの管理（1.3）

### 3.1 グループの概要と種類

#### 🔰 グループとは何か

**グループ（Google Groups）** は、複数のユーザーをひとまとめにして管理するための機能です。グループには以下の2つの大きな役割があります：

1. **コミュニケーション（メール配信）**: グループアドレスに送信されたメールを全メンバーに配信
2. **アクセス制御（権限管理）**: グループ単位でドライブ、カレンダー、その他リソースの権限を付与

---

### 3.2 グループ構造の設計

#### 試験で問われる4種類のグループ

| グループタイプ | 主な用途 | 特徴 |
| ------------- | --------- | ------ |
| **配信リスト（Distribution List）** | メール一斉送信 | グループアドレス宛のメールを全メンバーに転送 |
| **Collaborative Inbox（共有メールボックス）** | チームでのメール対応 | メールの割り当て・未解決/解決済み管理が可能 |
| **セキュリティグループ（Security Group）** | アクセス制御・IAM | Cloud リソースへのアクセス権限付与に使用 |
| **動的グループ（Dynamic Group）** | 条件ベースの自動メンバー管理 | LDAP クエリ条件でメンバーを自動追加・削除 |

---

### 3.3 配信リストの作成と管理

#### 配信リストの作成手順

```
Admin コンソール → ディレクトリ → グループ → グループを作成
```

**設定項目**:

- グループ名（例：`営業部全員`）
- グループのメールアドレス（例：`sales-all@example.com`）
- グループの説明
- アクセスの種類（組織内のみ / 外部からも送信可能 など）

#### グループのアクセス設定

| 設定 | 説明 |
| ------ | ------ |
| **誰がメンバーになれるか** | 組織内のユーザーのみ / 外部ユーザーも可 / 招待制 |
| **誰がグループにメール送信できるか** | グループのオーナー / メンバー / 組織内全員 / 全員（外部含む）|
| **グループの投稿を閲覧できるのは** | オーナー / メンバー / 組織内全員 / 全員 |
| **グループに参加できるのは** | 招待のみ / 組織内のユーザーが自由に参加 |

---

### 3.4 Collaborative Inbox（共有メールボックス）

#### 🔰 Collaborative Inbox とは

**Collaborative Inbox** は、チームで共同してメールに対応するための機能です。カスタマーサポート、ヘルプデスク、共有メールアドレス（info@, support@ など）の管理に最適です。

```
【通常の配信リストとの違い】

配信リスト:
  送信者 → グループアドレス → 全メンバーに転送されるだけ

Collaborative Inbox:
  送信者 → グループアドレス → キューに蓄積
                              ↓
              担当者に【割り当て（Assign）】
              ステータス管理【未解決 / 解決済み】
              チームメンバーが協力して対応
```

#### Collaborative Inbox の設定手順

```
Admin コンソール → グループ → [対象グループ] → 
グループの設定 → 会話をメンバーに割り当て可能にする → 有効化
```

**権限設定**（グループメンバーに付与できる役割）:

- **オーナー**: 全機能を管理
- **マネージャー**: メンバー管理とモデレーション
- **メンバー**: 通常の使用（割り当て・ステータス変更は設定次第）

#### ✅ ベストプラクティス：Collaborative Inbox

- グループメンバー全員に **「割り当ておよびステータス変更」** 権限を付与する
- グループアドレスへのメールを受信したら、**24時間以内に担当者に割り当てる**運用ルールを策定する
- 未解決のメールを定期的にレビューする仕組みを作る

---

### 3.5 動的グループ（Dynamic Groups）

#### 🔰 動的グループとは

**動的グループ** は、ユーザーの属性（部署、役職、所属 OU など）に基づいて、メンバーシップを**自動的に維持**するグループです。

```
【動的グループの仕組み】

条件定義:
  「user.department == 'Engineering' && user.location == 'Tokyo'」
            ↓ Google が定期的にユーザー属性を評価
  東京在住のエンジニアが自動的にメンバーに追加される
            ↓ 属性が変わったら（異動・転勤）
  自動的にメンバーから削除される

管理者の手動作業は不要！
```

#### 動的グループで使用できる条件の例

| 条件の属性 | 例 |
| ----------- | ----- |
| **部署（department）** | `user.department == "Marketing"` |
| **所属 OU（orgUnitPath）** | `user.orgUnitPath == "/Japan/Tokyo"` |
| **役職（title）** | `user.title == "Manager"` |
| **従業員タイプ（employmentType）** | `user.customAttributes.employeeType == "FTE"` |
| **場所（location）** | `user.location.name == "Tokyo HQ"` |

#### 動的グループの作成手順

```
Admin コンソール → ディレクトリ → グループ → グループを作成 
→ 「メンバーシップを動的にする」を選択
→ クエリを入力（例：user.department == "Sales"）
→ グループを保存
```

> ⚠️ **注意**: 動的グループは **Enterprise エディション** または **Cloud Identity Premium** が必要です。
> 🔗 **参考（動的グループ）**: https://support.google.com/a/answer/11192679

---

### 3.6 セキュリティグループ

#### セキュリティグループの目的

**セキュリティグループ** は、Google Cloud の IAM や Google Workspace のアクセス制御に使用する特別なグループです。

```
【セキュリティグループの主な用途】

1. Google Cloud IAM でのロール付与
   例: "全 Cloud 管理者" グループに roles/viewer を付与

2. コンテキストアウェアアクセスポリシーの適用
   例: "特権アクセスグループ" のみ管理コンソールへのアクセスを許可

3. Google Workspace のアクセス制御
   例: "機密ドキュメントアクセスグループ" のみ特定の共有ドライブにアクセス可能
```

#### セキュリティグループの有効化

```
Admin コンソール → グループ → [対象グループ] → 
グループ情報 → セキュリティグループとして宣言する
```

> ⚠️ **重要**: セキュリティグループは一度設定すると、**通常のグループに戻すことができません**。

#### ✅ ベストプラクティス：グループ管理

| # | ベストプラクティス |
| --- | ------------------- |
| 1 | **グループの命名規則を統一する**（例：`team-<部署名>-<目的>`）|
| 2 | **グループにオーナーを必ず設定する**（定期的にメンバーレビューを実施してもらう）|
| 3 | **大きなグループ（全社員など）の外部メール受信を制限する**（スパム・誤送信リスクを軽減）|
| 4 | **動的グループを活用してメンバー管理を自動化する**（手動管理の漏れを防ぐ）|
| 5 | **セキュリティグループへの直接アクセス付与を避け、ロールベースで管理する** |

> 🔗 **参考（グループ管理）**: https://support.google.com/a/answer/33329  
> 🔗 **参考（セキュリティグループ）**: https://support.google.com/a/answer/10607394

## Chapter 4: ドメインの管理（1.4）

### 4.1 ドメインの種類と役割

#### 🔰 ドメインの基本概念

Google Workspace では、複数のドメインを管理することができます。それぞれの種類と役割を正確に理解することが重要です。

| ドメイン種別 | 説明 | メールアドレスの例 |
| ------------ | ------ | ----------------- |
| **プライマリドメイン** | 最初に登録したメインドメイン | taro@example.com |
| **セカンダリドメイン** | 追加した独立したドメイン | taro@example.co.jp |
| **ドメインエイリアス** | プライマリドメインの別名（同じ受信箱） | taro@example.jp → taro@example.com の受信箱に届く |

---

### 4.2 ドメインの追加と確認

#### セカンダリドメインの追加手順

```
Step 1: Admin コンソール → アカウント → ドメイン → ドメインを管理 → ドメインを追加

Step 2: ドメイン名を入力して「続行してドメインの所有権を証明」をクリック

Step 3: DNS レコードの追加（以下のいずれか）
   ├── TXT レコード（推奨）
   │   例: google-site-verification=xxxxxxxxxxxxxxxxxxxx
   │   DNS に TXT レコードとして追加
   │
   └── CNAME レコード
       例: xxxxx.example.com → gv-xxxxxx.dv.googlehosted.com

Step 4: DNS の変更が伝播するまで待機（最大48時間、通常は数時間以内）

Step 5: Admin コンソールで「確認」をクリック
```

#### TXT レコード確認の方法

```bash
# Linux / Mac での確認方法
nslookup -type=TXT example.com
# または
dig TXT example.com

# Windows PowerShell での確認方法
Resolve-DnsName -Name example.com -Type TXT
```

---

### 4.3 セカンダリドメイン vs ドメインエイリアス

#### 重要な違いの理解

| 比較項目 | セカンダリドメイン | ドメインエイリアス |
| --------- | ---------------- | ---------------- |
| **独立したユーザー作成** | ✅ 可能（taro@example.co.jp を独立して作成） | ❌ 不可 |
| **受信箱** | 独立した受信箱 | プライマリドメインと同じ受信箱を共有 |
| **送信アドレス** | そのドメインのアドレスから送信可能 | そのドメインのアドレスから送受信可能（参考: support.google.com/a/answer/7502379） |
| **主な用途** | 別ブランド・別地域のメールアドレス | 既存アドレスで別名（エイリアス）を用いた送受信をしたい場合 |
| **ライセンス消費** | ユーザーごとにライセンスが必要 | 追加ライセンス不要（既存ユーザーに自動適用） |

```
【具体例で理解する】

シナリオ: example.com と example.co.jp の両方を運営

A. ドメインエイリアスとして設定:
   taro.yamada@example.com（プライマリ）
   taro.yamada@example.co.jp → 同じ受信箱に届く（追加ライセンス不要）

B. セカンダリドメインとして設定:
   taro.yamada@example.com（example.com のユーザー）
   taro.yamada@example.co.jp（別途作成したユーザー → 別ライセンス必要）
   ※ 完全に独立した受信箱を持てる
```

---

### 4.4 プライマリドメインの変更

```
Admin コンソール → アカウント → ドメイン → ドメインを管理 → 
プライマリとして設定
```

> ⚠️ **重要な注意点**: プライマリドメインを変更すると、すべてのユーザーの **メインメールアドレスのドメイン部分が変わります**。事前にユーザーへの通知と十分な計画が必要です。

#### ✅ ベストプラクティス：ドメイン管理

| # | ベストプラクティス |
| --- | ------------------- |
| 1 | **DNS への TXT レコード追加でドメインを確認する**（CNAME より影響範囲が小さい）|
| 2 | **ドメイン確認後は DNS レコードを削除しない**（削除すると確認が取り消される場合がある）|
| 3 | **不要なドメインは Admin コンソールから削除する**（セキュリティリスクの最小化）|
| 4 | **ドメイン追加前に MX レコードを確認し、メール配信への影響を把握する** |

> 🔗 **参考（ドメイン追加）**: https://support.google.com/a/answer/9008873  
> 🔗 **参考（ドメインエイリアス）**: https://support.google.com/a/answer/7502379

## Chapter 5: 建物とリソースの管理（1.5）

### 5.1 建物とリソースとは

#### 🔰 カレンダーリソースの概念

**カレンダーリソース**は、予約・スケジュール管理が必要な物理的なもの（会議室、備品など）を Google Calendar に登録したものです。ユーザーがカレンダーで会議を作成する際に、これらのリソースを予約できます。

```
【リソースの種類例】

場所系リソース:
  ├── 会議室（Meeting Room）
  ├── トレーニングルーム
  └── 電話ブース

備品系リソース:
  ├── プロジェクター
  ├── ビデオ会議システム
  ├── 社用車
  └── 貸出 PC
```

---

### 5.2 建物と部屋の設定構造

#### 階層構造の理解

```
建物（Building）
  └── フロア（Floor）
      └── 部屋・リソース（Room / Resource）
          └── フィーチャー（Features）
              ├── 定員（容量）
              ├── ビデオ会議設備の有無
              ├── ホワイトボードの有無
              └── バリアフリー対応
```

#### 建物の作成手順

```
Admin コンソール → ディレクトリ → 建物とリソース → 建物を追加
```

**設定項目**:

- 建物名（例: `東京本社ビル`）
- 建物 ID（一意の識別子）
- 住所（緯度・経度も設定可能）
- フロア数

---

### 5.3 リソースの一括作成（CSV インポート）

#### 大量のリソースを効率的に作成する方法

多くの会議室がある場合、一件ずつ手動で作成するのは非効率です。**CSV インポート**を使って一括作成できます。

**手順**:

```
Admin コンソール → ディレクトリ → 建物とリソース → 
リソース → ダウンロード（テンプレートを取得）→ 編集後にアップロード
```

**CSV フォーマット例**:

```csv
ID,Building ID,Floor Name,Floor Section,Resource Name,Capacity,Description,User Visible Description
room-101,tokyo-hq,1,A棟,会議室101,10,一般会議室 定員10名 プロジェクターあり,1F 10名収容 プロジェクター完備
room-201,tokyo-hq,2,A棟,会議室201,6,小会議室 定員6名,2F 6名収容
training-room,tokyo-hq,3,B棟,トレーニングルーム,30,研修室 定員30名 モニター6台あり,3F 30名収容
```

> 🔗 **参考（一括作成）**: https://support.google.com/a/answer/1423286

### 5.4 新しいリソースの作成と管理

#### リソースの作成手順（個別）

```
Admin コンソール → ディレクトリ → 建物とリソース → 
リソース → ➕（リソースを追加）
```

**設定項目詳細**:

| 項目 | 説明 | 例 |
| ------ | ------ | ----- |
| **カテゴリ** | リソースの種類 | 会議室 / 会議スペース / その他 |
| **リソース名** | 表示名 | `会議室A-10F（定員12名）` |
| **建物** | 所属する建物 | 東京本社ビル |
| **フロア** | 所属するフロア | 10F |
| **説明** | 内部管理用の説明 | 大型モニター、電話会議システム完備 |
| **ユーザー向け説明** | 予約時に表示される説明 | 10名から12名の会議に最適。事前にビデオ会議システムをテストしてください |

---

### 5.5 リソース予約権限の設定

#### 予約権限の3つのレベル

| 権限レベル | 説明 |
| ---------- | ------ |
| **無制限の予約** | 誰でも（外部ユーザーを除く）自由に予約可能 |
| **自動承認または手動承認** | 予約は自動承認されるか、オーナーが手動で承認する |
| **オーナーのみ予約可能** | リソースカレンダーのオーナーだけが予約できる |

#### 予約設定の詳細オプション

```
Admin コンソール → リソース → [リソース選択] → 予約設定
```

| 設定項目 | 説明 |
| --------- | ------ |
| **事前予約できる最大日数** | 何日前から予約できるかを制限（例：最大60日前）|
| **最低使用時間** | 最短何分の予約ができるか（例：最低30分） |
| **最大使用時間** | 1回の予約で最長何時間予約できるか（例：最大8時間）|
| **定期的な予約を許可** | 定期会議での予約を許可するか |
| **会議終了後の自動キャンセル** | 予約時間が来ても参加者がいない場合に自動キャンセル（Room Insights 連携）|

### 5.6 フィーチャー（Features）の設定

#### フィーチャーとは

**フィーチャー** は、リソースの特徴・設備情報を登録するための機能です。ユーザーが会議室を検索する際の絞り込み条件として活用されます。

#### フィーチャーの作成手順

```
Admin コンソール → ディレクトリ → 建物とリソース → 
フィーチャーの管理 → フィーチャーを追加
```

**一般的なフィーチャーの例**:

| フィーチャー名 | 説明 |
| ------------- | ------ |
| `プロジェクター` | プロジェクターが設置されている |
| `大型モニター` | 大型ディスプレイが設置されている |
| `ビデオ会議システム` | Meetのハードウェアデバイスが設置されている |
| `ホワイトボード` | ホワイトボードが設置されている |
| `電話会議システム` | 専用電話会議端末がある |
| `バリアフリー` | 車椅子でもアクセス可能 |
| `自然光あり` | 窓あり |

#### フィーチャーをリソースに追加する

```
Admin コンソール → リソース → [リソース選択] → 
フィーチャーを追加 → 作成済みフィーチャーを選択
```

#### ✅ ベストプラクティス：建物とリソース管理

| # | ベストプラクティス |
| --- | ------------------- |
| 1 | **リソース名に検索しやすい情報を含める**（フロア・定員・特徴など）|
| 2 | **フィーチャーを標準化して登録する**（全リソースで統一されたフィーチャーを使用）|
| 3 | **適切な予約権限を設定する**（全社員が全リソースを自由予約できないよう制限）|
| 4 | **リソースの説明は管理者向けとユーザー向けを使い分ける** |
| 5 | **予約できる最大日数を設定し、無制限予約による占有を防ぐ** |
| 6 | **一括 CSV インポートを活用し、50以上のリソース作成を効率化する** |

> 🔗 **参考（建物とリソース）**: https://support.google.com/a/answer/1686462

## Chapter 6: Section 1 試験対策まとめ

### 6.1 出題パターン別の重要ポイント

#### パターン①：移行ツールの選定

```
【典型的な問題文の例】
「現在 Microsoft Exchange オンプレミス（社員数500名）を使用している。
 メール・カレンダー・連絡先のデータを Google Workspace に移行したい。
 最適なツールはどれか？」

【考え方】
- 大規模（多機能、複雑な要件）→ Google Workspace Migrate
- 中規模（標準的な Exchange 移行）→ GWMME
- 小〜中規模（GUI で操作、シンプルな要件）→ データ移行サービス
- ユーザー情報の同期（メールデータではない）→ GCDS

【重要な区別】
GCDS = メールデータの移行ではなく、ユーザー/グループ情報の同期！
```

#### パターン②：アカウント状態の選択

```
【典型的な問題文の例】
「退職者のアカウントについて、法的要件でデータを2年間保持する必要がある。
 コストを最小化しながらコンプライアンスを満たすにはどうするか？」

【正解のアプローチ】
1. アカウントを【停止】（即時アクセスをブロック）
2. ライセンスを通常版から【Archived User】に変更（コスト削減）
3. Google Vault で保持ルールを設定（2年間のデータ保持）

【よくある誤答】
❌ 削除する → 20日後にデータが消える
❌ 通常ライセンスのまま停止 → コストが高い
```

#### パターン③：OU 設計の問題

```
【典型的な問題文の例】
「財務部門は Drive の外部共有を禁止し、
 それ以外の部門はデフォルトで外部共有を許可したい。
 最も適切な OU 設計はどれか？」

【正解のアプローチ】
- ルート組織: 外部共有【許可】（デフォルト）
- 財務部 OU: 外部共有【禁止】（上書き）

【OU 設計の基本原則】
「同じポリシーが適用されるユーザーは同じ OU に」
「例外が必要なグループは別 OU に分ける」
```

#### パターン④：グループタイプの選択

```
【典型的な問題文の例】
「カスタマーサポートチームが support@example.com 宛のメールを
 チームで管理したい。メールの担当者割り当てと解決済み管理が必要。」

【正解】
Collaborative Inbox → メールの割り当て・ステータス管理ができる

【各グループの使い分けポイント】
配信リスト → メールを全員に一斉送信するだけでいい場合
Collaborative Inbox → チームで担当分けして対応したい場合
動的グループ → 属性ベースで自動メンバー管理したい場合
セキュリティグループ → IAM やアクセス制御で使いたい場合
```

#### パターン⑤：ドメイン種別の選択

```
【典型的な問題文の例】
「example.com に加えて example.co.jp も使いたい。
 既存ユーザーが両方のアドレスでメールを受信できるようにしたい。
 追加のライセンスコストは発生させたくない。」

【正解】
ドメインエイリアスとして example.co.jp を追加
→ 既存ユーザーに自動的に @example.co.jp アドレスも付与される
→ 追加ライセンス不要

【セカンダリドメインとドメインエイリアスの使い分け】
セカンダリドメイン → 独立したユーザーが必要な場合（ライセンス必要）
ドメインエイリアス → 既存ユーザーが別アドレスでも受信したい場合（ライセンス不要）
```
